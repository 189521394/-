# 论文写作风格规范

本文件记录论文正文（论文大纲.md）的写作格式约定，确保各章节风格统一，也方便后续复制到 docx 时减少手动调整。

## 1. 禁用 Markdown 加粗语法

全文不使用 `**文字**` 格式的加粗标记。因为内容最终要复制进 docx，Markdown 加粗符号在 Word 中不会自动转换，反而增加清理工作量。

小标题（如 3.3.1）直接用纯文本数字编号，不用加粗。

## 2. 代码与解释分离

每节的正文解释与代码块分开摆放：

- 正文各子节（如 3.3.1 ~ 3.3.4）为纯文字分析，不嵌入任何代码块。
- 最后一子节（如 3.3.5 完整代码）集中展示该文件的全部源码，代码直接从项目文件夹中原样复制，不添加注释、不改排版。

代码块用 Markdown 的 ` ```scala ` 或 ` ```java ` 围起来，方便在支持 Markdown 的编辑器中高亮。

## 3. 表格适度使用

表格不需要太多，只在确实比纯文字更清晰时才用。典型场景：集群角色分配、软件版本列表、HBase 表结构。

不用表格的场景：代码执行步骤、DataFrame 列说明（用文字描述即可）。

## 4. 第四章起新增规则

以下规则仅对第四章及之后的新增内容生效，前三章已完成部分不做追溯修改：

- 正文中不使用 Markdown 的 `-` 无序列表符号。需要分点说明时，用中文序号（1）（2）（3）或自然段过渡。
- 正文中不使用反引号 `` ` `` 包裹代码级文本（如类名、方法名、变量名）。直接写中文描述或自然嵌入即可，例如 "通过 spark.read.json 方法加载数据"而非 "通过 `spark.read.json()` 方法加载数据"。

## 5. 插图处理

需要插图时，不在 md 文件中嵌入图片。做法是：正文中标注 "如图 X-X 所示"，然后在对话中输出插图的详细文字描述（元素位置、箭头方向、标注文字），由人工根据描述绘制并插入 docx。

## 6. 命名约定

- 代码文件、类名、方法名、变量名保留原文（英文），不翻译。
- 技术术语首次出现时用中文解释，后续直接用术语。例如首次写 "窗口函数（Window Function）"，后续写 "窗口函数"。

## 7. 文件结构

每节末尾统一加一个 "X.X.X 完整代码" 子节，放该文件的全部源码。
