# 电商平台数据分析系统 —— 论文大纲

## 摘要

随着电子商务的快速发展，电商平台积累了海量用户行为数据。如何从这些数据中提取有价值的商业洞察，已成为企业精细化运营的关键需求。本文设计并实现了一套电商平台数据分析系统，分为离线分析与 Web 可视化两个部分。离线分析部分基于 Spark SQL 对用户行为数据集进行处理，利用 DataFrame API 和窗口函数完成品类用户行为聚合、区域热销商品排名和页面跳转转化率三项分析，并将结果存入 HBase 分布式数据库；Web 可视化部分基于 Spring MVC 框架构建，从 HBase 读取分析结果，通过 ECharts 柱状图直观展示十大火爆品类、五十州热销商品分布和页面单跳转化率。系统打通了从数据离线分析到结果可视化展示的完整链路，为电商数据驱动决策提供了一套可行的技术方案。

**关键词：** Spark SQL；HBase；Spring MVC；ECharts；电商数据分析

## 第一章 绪论

### 1.1 选题背景与意义

近年来，全球电子商务市场持续高速增长。以 Amazon 为代表的电商平台每天产生数以亿计的用户行为数据，涵盖浏览、加购、购买、页面跳转等多类操作。这些数据中蕴含着用户消费偏好、商品市场热度和网站流量转化率等关键商业信息。然而，原始行为数据体量大、结构松散，传统的关系型数据库和单机分析工具在处理效率和存储扩展性方面均面临瓶颈，无法有效支撑大规模数据的离线统计与多维分析。

大数据技术的成熟为解决上述问题提供了可行路径。Spark SQL 作为 Spark 生态中的结构化数据处理引擎，提供了 DataFrame API 和窗口函数等高级抽象，能够以简洁的代码实现分组聚合、TopN 排名和转化率计算等复杂分析逻辑。HBase 作为构建于 Hadoop HDFS 之上的分布式列式数据库，具备高扩展性和灵活的稀疏列存储模型，适合存储结构各异的海量分析结果。Spring MVC 框架和 ECharts 可视化库则为分析结果的 Web 展示提供了工程化的前后端解决方案。

基于以上背景，本文设计并实现了一套电商平台数据分析系统。系统分为两大部分：第一部分基于 Spark SQL 对 Amazon 用户行为数据集进行离线分析，分别完成品类用户行为聚合、区域热销商品排名和页面跳转转化率三项统计，并将分析结果持久化至 Linux 集群的 HBase 数据库中；第二部分基于 Spring MVC 框架构建 Web 可视化端，从 HBase 读取数据并通过 ECharts 柱状图直观呈现数据洞察。系统打通了"本地分析 → 集群存储 → Web 展示"的完整数据处理链路，为电商平台的数据驱动运营提供了一套切实可行的技术方案。

## 第二章 相关技术介绍

2.1 Scala 语言概述

Scala 是一门运行于 JVM 之上的多范式编程语言，融合了面向对象与函数式编程的特性。其语法简洁，支持类型推断、模式匹配、高阶函数和不可变集合等特性，能够以较少的代码量表达复杂的数据处理逻辑。Scala 与 Java 无缝互操作，可直接调用 Java 类库。Spark 框架本身即采用 Scala 编写，因此 Scala 是 Spark 的原生开发语言，使用 Scala 编写 Spark 程序可获得最完整的 API 支持与编译时类型检查。本系统中三个离线分析程序（Amazon.scala、AmazonOrder.scala、Conversion.scala）以及 HBase 连接工具类（HBaseUtil.scala）均以 Scala 编写，并在其中定义了 Info、Product、conBean 三个 case class 用于封装分析结果。

2.2 Spark SQL 与 DataFrame API

Spark SQL 是 Apache Spark 生态中用于结构化数据处理的模块，提供了 DataFrame API 和 SQL 查询两种编程接口。DataFrame 是一种按命名列组织的分布式数据集合，可类比于关系数据库中的表，底层由 Catalyst 优化器负责查询优化与执行计划生成。

DataFrame API 提供了丰富的函数式操作：filter 用于条件过滤，groupBy 与 agg 用于分组聚合，join 用于多表连接，orderBy 用于排序，select 用于列投影。此外，Spark SQL 提供了窗口函数（Window Function），可在不缩减行数的前提下完成分组内排名和跨行引用等高级计算，常用的窗口函数包括 row_number()（分组内行号）和 lead()（获取后一行值）。

本系统的三项分析任务均基于 Spark SQL 实现：品类行为分析（Amazon.scala）利用 DataFrame 的 filter、groupBy、agg 和 join 操作，按品类统计浏览、加购、购买三种行为次数并取 Top10；区域热销商品分析（AmazonOrder.scala）使用 Window.partitionBy 结合 row_number() 窗口函数，按地区分组取各商品销量前三名；页面单跳转化率分析（Conversion.scala）使用 lead() 窗口函数计算用户页面跳转路径，再通过 join 和除法运算得出每一步的转化率。

2.3 HBase 分布式数据库

HBase 是构建于 Hadoop HDFS 之上的分布式列式数据库，具备高扩展性、稀疏存储和随机读写能力，适合存储海量结构化与半结构化数据。本系统将 Spark SQL 的三项分析结果存入 HBase，再由 Web 端从中读取展示，HBase 在系统中承担"分析结果持久化存储"的中枢角色。

2.3.1 HBase 数据模型与架构

HBase 的数据模型由表（Table）、行键（RowKey）、列族（Column Family）、列限定符（Column Qualifier）和时间戳（Timestamp）五层组成。每张表按行键字典序排序存储，列族下的列可动态扩展，空列不占存储空间。本系统三张分析结果表的列族均命名为 info：categoryCount 表以 category_id 为行键，存储 view/cart/purchase 三列；top3 表以 address_name 为行键，存储 pro5115/pro4856/pro4767 三列；conversion 表以 jump 路径字符串为行键，存储 conversion 列。

在架构层面，HBase 底层依赖 HDFS 实现数据持久化，表按行键范围横向切分为多个 Region，每个 Region 由一个 RegionServer 负责读写。HMaster 负责 Region 分配与负载均衡，ZooKeeper 负责集群元数据存储与节点协调。本系统 Linux 集群部署了三节点 ZooKeeper（hadoop1/2/3:2181），HBase 客户端通过 ZooKeeper 获取集群连接信息。

2.3.2 HBase 集群角色（HMaster / RegionServer / ZooKeeper）

HMaster 是 HBase 集群的管理节点，负责表的创建与删除、Region 的分配与迁移以及 RegionServer 的故障恢复，不直接参与数据读写。RegionServer 是数据存储与读写的工作节点，每个 RegionServer 管理若干 Region，处理客户端的 Put、Get、Scan 等请求。ZooKeeper 为分布式协调服务，存储 HBase 元数据（如 ROOT 表位置），监控 RegionServer 的存活状态，并在 HMaster 故障时支持主备切换。

2.4 Spring MVC 框架

Spring MVC 是 Spring 生态中用于构建 Web 应用程序的轻量级框架，基于 MVC（Model-View-Controller）设计模式，通过 DispatcherServlet 前端控制器统一分发请求。其核心工作流程为：客户端请求首先到达 DispatcherServlet，经 HandlerMapping 定位到对应的 Controller 方法，Controller 调用业务逻辑层处理后返回 ModelAndView 或经 @ResponseBody 直接序列化为 JSON 响应。

本系统 sparkView 项目采用两级 Spring 上下文（父子容器）架构：Root WebApplicationContext（ac.xml）负责管理 HBase 连接组件（Connect）、数据访问对象（DAO）和业务服务（Service），Servlet WebApplicationContext（springMVC.xml）负责管理 Controller、启用注解驱动 MVC（`<mvc:annotation-driven>`）及静态资源映射。控制层由 myController 实现，通过 @Autowired 注入 Service，定义 /、/top10、/top3、/jump 四个路由，其中后三者标注 @ResponseBody 借助 Jackson 自动将返回的 Object[] 序列化为 JSON，供前端 ECharts 图表通过 AJAX 请求消费。

2.5 ECharts 与 jQuery 前端技术

ECharts 是百度开源的一款基于 JavaScript 的数据可视化图表库，支持柱状图、折线图、饼图等多种图表类型，提供标题、图例、工具箱（支持图表切换、数据视图、还原、保存为图片）等丰富的交互组件，且兼容主流浏览器。本系统使用 ECharts 4.3.0 绘制三张分组柱状图，分别展示十大火爆品类的用户行为数据、五十州热销商品分布和页面单跳转化率，通过 toolbox 组件支持柱状图与折线图的动态切换以及图表导出功能。

jQuery 是一个轻量级 JavaScript 库，简化了 DOM 操作、事件处理和 AJAX 异步请求。本系统使用 jQuery 1.11.3 的 $.ajax 方法向 Controller 的三个接口（/top10、/top3、/jump）发起 GET 请求，获取 JSON 数组后传入 echarts.setOption() 完成图表渲染。前端页面 index.jsp 采用 Flexbox 弹性布局，左侧深色侧边栏提供三个导航项，点击后通过 JavaScript 函数切换不同图表，并配合 CSS 动画实现导航选中高亮与悬停过渡效果。

2.6 Maven 项目管理工具

Maven 是 Apache 软件基金会推出的项目构建与依赖管理工具，基于项目对象模型（POM）描述项目的构建配置、依赖关系和插件信息。开发者只需在 pom.xml 中声明所需依赖的 groupId、artifactId 和 version，Maven 即可从中央仓库自动下载并管理依赖的传递性，省去手动导入 JAR 包的繁琐工作。

本系统两个子项目均基于 Maven 构建。session 项目依赖 spark-core_2.12 3.3.0、spark-sql_2.12 3.3.0、hbase-client 2.4.9 和 hbase-common 2.4.9，Scala 源码编译为 JAR 包后在 Linux 集群上通过 spark-submit 提交运行。sparkView 项目除 hbase-client/hbase-common 外，还依赖 spring-webmvc 5.2.8、jackson-* 2.13.4 以及 servlet/jsp-api，通过 mvn clean package 打包为 WAR 文件部署至 Tomcat 9 容器。Maven 统一管理了两个项目的所有第三方依赖版本，确保了构建过程的一致性与可复现性。

2.7 本章小结

本章对系统所涉及的核心技术进行了概要介绍。Scala 语言作为 Spark 的原生开发语言，以其函数式编程特性和简洁语法支撑了离线分析模块的实现；Spark SQL 提供了 DataFrame API 和窗口函数等结构化数据处理能力，是本系统三项分析任务的核心计算引擎；HBase 作为分布式列式数据库，承担了分析结果持久化存储与 Web 端数据读取的中枢角色；Spring MVC 框架以两级上下文架构实现了后端接口的工程化构建；ECharts 与 jQuery 共同完成了前端可视化展示与用户交互；Maven 则统一管理了项目依赖与构建流程。以上技术相互衔接，共同构成了"离线分析 → 集群存储 → Web 展示"的完整技术栈。

3.2 数据源说明

本系统使用两组 JSON 格式的用户行为数据集，分别用于品类/商品分析和转化率分析。

第一组数据集 `user_session.json` 为用户会话行为数据，每行为一条用户对商品的操作记录，核心字段包括：user_id（用户标识）、category_id（商品品类 ID）、product_id（商品 ID）、event_type（事件类型，取值为 view / cart / purchase）、address_name（用户所在州名）及 event_time（操作时间戳）。该数据集分别供 Amazon.scala 做品类维度的三种行为聚合统计，以及 AmazonOrder.scala 做各州分组的热销商品 Top3 排名。

第二组数据集 `user_conversion.json` 为用户页面跳转数据，核心字段包括：userid（用户标识）、pageid（当前页面 ID）和 actionTime（操作时间）。该数据集供 Conversion.scala 按用户分组、按时间排序后计算相邻页面的跳转路径及转化率。

两组数据均以 JSON 行格式（每行一个独立的 JSON 对象）存储在本地文件系统中，Spark SQL 通过 `spark.read.json()` 直接解析为 DataFrame，无需额外定义 Schema。数据文件路径在代码中硬编码为 `C:\data\user_session.json` 和 `C:\data\user_conversion.json`，部署时可根据实际环境修改。

3.3 品类用户行为分析（Amazon.scala）

3.3.1 分析目标与思路

品类用户行为分析是本系统第一项离线分析任务，目标是从海量用户行为数据中统计每个商品品类（category_id）在浏览（view）、加购（cart）和购买（purchase）三种行为上的用户参与量，并从中筛选出综合热度最高的前十个品类，即"十大火爆品类"。

分析的数据源为 `user_session.json` 数据集，每条记录包含 user_id、category_id、product_id、event_type、address_name 和 event_time 六个字段，其中 event_type 字段取值为 view、cart 或 purchase 三者之一。分析的基本思路分为五步：（1）使用 `spark.read.json()` 将 JSON 数据加载为 DataFrame；（2）分别按 category_id 对三种 event_type 做分组计数，得到 viewDF、cartDF、purchaseDF 三个中间 DataFrame；（3）以 category_id 为连接键，通过两次内连接将三者横向合并为一张包含四列的宽表 resultDF；（4）按 viewCount → cartCount → purchaseCount 三级降序排列，取前 10 行；（5）将 Top10 结果批量写入 HBase categoryCount 表，rowKey 为 category_id，在 info 列族下存储 view、cart、purchase 三个列。

该分析由 `Amazon.scala` 单文件实现，入口为 `Amazon.main()` 方法。程序创建 SparkSession 时指定 master 为 `local[*]`（使用本地所有可用 CPU 核心），适用于 Windows 开发环境下的本地调试与运行。

3.3.2 DataFrame 读取与分组聚合

程序通过 `spark.read.json("C:\\data\\user_session.json")` 将 JSON 数据集加载为 DataFrame。Spark SQL 的 JSON 数据源能够自动推断字段名与数据类型，解析后 DataFrame 的 Schema 包含 user_id（String）、category_id（String）、product_id（String）、event_type（String）、address_name（String）和 event_time（String）六个列，无需手动定义结构。加载完成后，程序依次执行两步预览操作：`dsdf.show()` 输出前 20 行原始数据以验证读取正确性；`dsdf.groupBy("category_id", "event_type").count().orderBy("category_id").show()` 按品类和事件类型交叉分组计数，输出数据分布的全貌，为后续独立聚合提供对照基准。

分组聚合是本节的核心步骤。程序对原始 DataFrame 分别按三种 event_type 进行过滤和计数：筛选 event_type 为 view 的行，按 category_id 分组后调用 `count("*")` 聚合，结果列取别名 viewCount，得到 `viewDF`；同理，筛选 cart 得到 `cartDF`（列别名 cartCount），筛选 purchase 得到 `purchaseDF`（列别名 purchaseCount）。三次聚合各自独立扫描原数据集一次。以上三次聚合的结果中，viewDF 包含 category_id 和 viewCount 两列，cartDF 包含 category_id 和 cartCount 两列，purchaseDF 包含 category_id 和 purchaseCount 两列。

由于三个 DataFrame 的分组键均为 category_id，Spark Catalyst 优化器在执行物理计划时可对公共子表达式进行复用。对于中等规模（十万至百万级行数）的本地数据集，该方案在代码清晰度与运行性能之间取得了合理的平衡。

3.3.3 多表 Join 与 Top10 筛选

三个聚合结果各自只包含 category_id 加一种行为的计数，需横向合并。程序使用 DataFrame 的 join 操作，以 category_id 为连接键，采用内连接（inner join）依次合并 viewDF、cartDF 和 purchaseDF。`Seq("category_id")` 表示连接键列名，当两张表键列名相同时 Spark 自动去重。内连接仅保留在三个 DataFrame 中均存在的 category_id——若某品类缺少某种行为类型将被排除，这符合"热度由三种行为共同构成"的分析语义。合并后的 resultDF 包含四列：category_id、viewCount、cartCount、purchaseCount。

Top10 筛选采用三级排序规则：以 viewCount 降序为首要排序键（浏览反映品类整体曝光度，数据量最大），cartCount 和 purchaseCount 降序为次要和第三键（在浏览量相同时区分转化深度）。排序后调用 `take(10)` 将前 10 行从 Executor 收集到 Driver 端，返回 `Array[Row]`，再通过 map 将每行映射为 Info case class 实例。Info 定义于 HBaseUtil.scala，包含 id（String）、view（Long）、cart（Long）和 purchase（Long）四个字段，作为分析结果在内存中的类型化载体，便于后续 HBase 批量写入时的类型安全传递。

3.3.4 写入 HBase categoryCount 表

Top10 筛选完成后，程序调用 HBaseUtil 工具类执行写入操作，分为三步：

（1）初始化连接：实例化 HBaseUtil 并调用 `init()` 方法，内部通过 `HBaseConfiguration.create()` 创建配置对象，设置 ZooKeeper Quorum 地址为 `hadoop1:2181,hadoop2:2181,hadoop3:2181`，然后调用 `ConnectionFactory.createConnection(conf)` 建立与 HBase 集群的长连接，保存为成员变量供后续操作复用。

（2）建表：调用 `createTable("categoryCount")` 创建分析结果表。方法内部获取 Admin 接口，构造 TableDescriptorBuilder 并设置列族名为 info（字节数组形式）。若表已存在（如重复运行分析），先 disable 再 delete 旧表，确保每次写入的是最新结果。

（3）批量写入：调用 `infoPut("categoryCount", top10.toIndexedSeq)` 将 Top10 序列批量写入。该方法遍历每个 Info 对象构造 HBase Put 实例：以 category_id 的 UTF-8 字节为 rowKey，在 info 列族下分别添加 view、cart、purchase 三列，列值为 Long 型计数的 8 字节大端序编码。所有 Put 对象收集为 Java List 后通过 `table.put(puts)` 一次性提交至 RegionServer，利用 HBase 的批量写入机制减少网络往返。写入完成后依次关闭 Table 和 Connection。

categoryCount 表以 category_id 为 rowKey，info 为列族，其下列 view、cart、purchase 三个列限定符，分别存储 Long 型的三种行为计数。

分析结果存入 HBase 后，由第四章的 sparkView Web 可视化端通过 DAO 层发起 Scan 全表扫描，读取全部 10 行记录并以分组柱状图形式呈现在前端页面。

3.3.5 完整代码

Amazon.scala 的完整实现如下：

```scala
package example

import org.apache.spark.SparkContext
import org.apache.spark.sql.{DataFrame, SparkSession}
import org.apache.spark.sql.functions._

object Amazon {
    def main(args: Array[String]): Unit = {
        val spark: SparkSession = SparkSession.builder().appName("Amazon").master("local[*]").getOrCreate()
        val sc: SparkContext = spark.sparkContext
        val dsdf: DataFrame = spark.read.json("C:\\data\\user_session.json")
        dsdf.show()
        dsdf.groupBy("category_id", "event_type").count().orderBy("category_id").show()
        val viewDF: DataFrame = dsdf.filter(col("event_type") === "view").groupBy("category_id").agg(count("*").alias("viewCount"))
        val cartDF: DataFrame = dsdf.filter(col("event_type") === "cart").groupBy("category_id").agg(count("*").alias("cartCount"))

        hb.close()
```

3.4 区域热门商品分析（AmazonOrder.scala）

3.4.1 分析目标与思路

区域热门商品分析的目标是从各州（address_name）的用户行为数据中，分别找出销量最高的前三款商品（Top3），再从所有州的 Top3 结果中提取三款特定商品（product_id 分别为 1005115、1004856、1004767）在各州的销量，最终形成一张"州 × 三款商品"的交叉销量表。

与品类分析不同，此项分析的核心挑战在于"分组内排序取 TopN"——需要在每个 address_name 分组内部按销量降序排列并截取前三名。传统 SQL 中的 `GROUP BY` 聚合无法直接保留组内多条记录，Spark SQL 提供的窗口函数（Window Function）恰好解决了这一问题。窗口函数在 `GROUP BY` 的基础上为每一行计算一个窗口内序号，不缩减行数，从而可以在保留全部行的同时标记出组内排名。

分析流程分为四步：（1）按 (address_name, product_id) 分组统计每个商品在各州的销量；（2）定义窗口规范 `Window.partitionBy("address_name").orderBy(col("product_count").desc)`，以州为分区键、销量降序为排序键，使用 `row_number()` 窗口函数为每个分区内的行分配从 1 开始的行号；（3）过滤行号 ≤ 3 得到各州 Top3，再从中分别筛选三款特定商品，以 address_name 为键将三张子表内连接合并；（4）将结果写入 HBase top3 表。

该分析由 `AmazonOrder.scala` 单文件实现，入口为 `AmazonOrder.main()` 方法。

3.4.2 窗口函数 row_number() 实现分组 TopN

程序首先通过 `spark.read.json("C:\\data\\user_session.json")` 加载数据集（与 3.3 节使用同一数据源），然后按 (address_name, product_id) 双维度分组，使用 `count("*")` 聚合得到每个商品在每个州的销量，结果列别名为 product_count。此步骤将原始的用户级行为记录汇总为商品级销量统计。

接下来定义窗口规范 WindowSpec：`Window.partitionBy("address_name")` 指定分区键为州名，DataFrame 按州名被划分为若干互不重叠的分区，每个分区包含该州全部商品的销量记录；`.orderBy(col("product_count").desc)` 指定分区内按销量降序排列。将窗口规范应用于 `row_number()` 函数，通过 `withColumn("index", row_number().over(spec))` 为每个分区内的每一行生成从 1 开始的递增整数序号，销量最高的行序号为 1，次高为 2，以此类推。随后 `filter(col("index") <= 3)` 保留每个州销量前三的商品行。最终得到的 resultTemp 包含 address_name、product_id、product_count 和 index 四列，按 address_name 字典序排列。至此前两步——分组聚合与窗口排序截取——全部完成。

3.4.3 三款特定商品数据提取与合并

各州 Top3 结果中包含大量不同的 product_id，不利于横向对比。分析需求进一步要求聚焦于三款特定商品——1005115、1004856 和 1004767——观察它们在全美五十州的销量分布。程序从 resultTemp 中分别过滤出三款商品的数据：对每款商品，筛选 `product_id` 等于目标值，仅保留 address_name 和 product_count 两列，并通过 `toDF` 重命名 product_count 为对应商品的标识列名（prod5115、prod4856、prod4767），以便后续合并时区分列名。随后以 address_name 为连接键，通过两次内连接将三张子表合并为一张宽表 result，包含 address_name、prod5115、prod4856、prod4767 四列。内连接确保仅保留三款商品均有销售记录的州——若某州缺少其中任一款商品的数据，该州将从结果中剔除。

3.4.4 写入 HBase top3 表

与 3.3 节不同，本节写入 HBase 采用了 foreachPartition 模式。程序首先在主线程中创建 HBaseUtil 实例，完成建表操作（创建 top3 表，列族 info），然后关闭该实例。随后调用 `result.rdd.foreachPartition`，将 DataFrame 转换为 RDD，按分区并行写入 HBase：每个分区在各自的 Executor 上独立创建 HBaseUtil 实例和 Connection，遍历该分区内所有行，将每行数据封装为 Product case class 实例（包含 addName、pro5115、pro4856、pro4767 四个字段），调用 `productPut` 方法逐条写入 HBase，分区处理完毕后关闭各自的连接。

foreachPartition 的优势在于每个分区独立持有 Connection，避免了 Driver 端单连接写入的瓶颈，也避免了将全部数据收集到 Driver 端再逐条写入的内存压力，适合数据量较大的场景。top3 表以 address_name（州名）为 rowKey，info 列族下存储 pro5115、pro4856、pro4767 三列，列值为对应商品在该州的销量（Long 型）。

3.4.5 完整代码

AmazonOrder.scala 的完整实现如下：

```scala
package example

import org.apache.spark.SparkContext
import org.apache.spark.sql.expressions.{Window, WindowSpec}
import org.apache.spark.sql.functions._
import org.apache.spark.sql.{DataFrame, Dataset, Row, SparkSession}

object AmazonOrder {
    def main(args: Array[String]): Unit = {
        val spark: SparkSession = SparkSession.builder().appName("Amazon").master("local[*]").getOrCreate()
        val sc: SparkContext = spark.sparkContext
        val dsdf: DataFrame = spark.read.json("C:\\data\\user_session.json")

        val spec: WindowSpec = Window.partitionBy("address_name").orderBy(col("product_count").desc)

        val resultTemp: Dataset[Row] = dsdf.groupBy("address_name", "product_id").agg(count("*").alias("product_count"))
            .orderBy("address_name").withColumn("index", row_number().over(spec))
            .filter(col("index") <= 3)

        val df5115: DataFrame = resultTemp.filter(col("product_id") === "1005115").select("address_name", "product_count").toDF("address_name", "prod5115")
        val df4856: DataFrame = resultTemp.filter(col("product_id") === "1004856").select("address_name", "product_count").toDF("address_name", "prod4856")
        val df4767: DataFrame = resultTemp.filter(col("product_id") === "1004767").select("address_name", "product_count").toDF("address_name", "prod4767")

        val result: DataFrame = df5115.join(df4856, Seq("address_name"), "inner").join(df4767, Seq("address_name"), "inner")

        val adminUtil = new HBaseUtil
        adminUtil.init()
        adminUtil.createTable("top3")
        adminUtil.close()

        result.rdd.foreachPartition { item =>
            val util = new HBaseUtil
            util.init()
            item.foreach { row =>
                val add: String = row.getAs[String]("address_name")
                val pro5115: Long = row.getAs[Long]("prod5115")
                val pro4856: Long = row.getAs[Long]("prod4856")
                val pro4767: Long = row.getAs[Long]("prod4767")
                val prod = new Product(add, pro5115, pro4856, pro4767)
                util.productPut("top3", prod)
            }
            util.close()
        }
    }
}
```

3.5 页面单跳转化率分析（Conversion.scala）

3.5.1 分析目标与思路

页面单跳转化率分析的目标是计算用户在网站各页面之间的跳转转化率。所谓"单跳"（single jump），是指用户从页面 A 直接跳转到下一个页面 B 的一次相邻页面访问行为，表示为"A->B"路径。转化率的定义为：在所有访问了页面 A 的会话中，下一步跳转到页面 B 的比例，即 `跳转次数(A->B) / 页面A的总访问次数`。

该分析的数据源为 `user_conversion.json` 数据集，每条记录包含 userid（用户标识）、pageid（当前页面 ID）和 actionTime（操作时间）三个字段。分析的核心挑战在于"跨行计算"——每条记录只包含当前页面 ID，要构造 A->B 跳转路径，需要获取同一用户的下一条记录的 pageid 作为"下一页"。Spark SQL 的 lead() 窗口函数恰好满足这一需求：它可以在当前行中引用同一窗口内后第 N 行的值，配合按 actionTime 排序，即可为每行计算出下一页面的 ID。

分析流程分为四步：（1）使用 lead() 窗口函数按用户分组、按时间排序，为每条记录计算下一页 pageid，拼接为 jump 路径字符串；（2）分别统计每个页面（pageid）的总访问次数和每条跳转路径（jump）的总次数；（3）将两个统计结果 join，通过除法计算每条跳转路径的转化率；（4）将结果写入 HBase conversion 表。

该分析由 `Conversion.scala` 单文件实现，入口为 `Conversion.main()` 方法。

3.5.2 lead() 窗口函数计算跳转路径

程序首先通过 `spark.read.json("C:\\data\\user_conversion.json")` 加载页面跳转数据集，并调用 `dsdf.show()` 预览原始数据。随后定义窗口规范 WindowSpec：`Window.partitionBy(col("userid")).orderBy(col("actionTime"))`，即以用户 ID 为分区键、操作时间为排序键。这意味着每个用户的所有页面访问记录按时间先后排列，构成该用户的浏览轨迹序列。

在窗口规范的基础上，程序依次添加三个计算列。第一步，`withColumn("index", row_number().over(spec))` 使用 row_number() 为每个用户的访问记录按时间顺序分配序号，便于追溯每条记录在其浏览轨迹中的位置。第二步，`withColumn("pageNext", lead(col("pageid"), 1).over(spec))` 使用 lead() 窗口函数获取同一用户下一条记录的 pageid 值：lead() 的第一个参数为目标列（pageid），第二个参数为偏移量（1 表示下一行）。对于每个用户的最后一条记录，由于不存在下一行，lead() 返回 null。第三步，`withColumn("jump", concat_ws("->", col("pageid"), col("pageNext")))` 使用 concat_ws() 函数将当前页面 ID 和下一页 ID 以"->"分隔符拼接为跳转路径字符串，如"page1->page2"。对于最后一条记录，pageNext 为 null，concat_ws 会忽略 null 值，因此 jump 列将仅包含单个 pageid，后续通过过滤移除。

至此，conversionDF 包含原始三列加上 index、pageNext、jump 共六列。窗口函数的计算依托于 Spark 的 partition+sort 机制：数据先按 userid 做 shuffle 分区，每个用户的数据按 actionTime 排序后，row_number() 和 lead() 在各用户分区内独立计算，互不干扰。

3.5.3 转化率计算公式与实现

转化率计算需要两个聚合统计量作为分子和分母。程序首先从 conversionDF 出发，分别做两次分组聚合：

- pageDF：按 pageid 分组，`count("*")` 统计每个页面作为当前页出现的总次数，列别名为 pageSumCount。这作为转化率的分母——表示"有多少次用户访问了该页面"。
- jumpDF：按 jump 分组，`count("*")` 统计每条跳转路径出现的总次数，列别名为 jumpSumCount。这作为转化率的分子——表示"从页面 A 跳转到页面 B 的次数"。

随后将 pageDF、conversionDF、jumpDF 三者通过两次内连接合并：以 pageid 为键连接 pageDF 与 conversionDF，再以 jump 为键连接 jumpDF。合并后通过 `withColumn("conversion", col("jumpSumCount").cast("Double") / col("pageSumCount"))` 计算转化率：分子 jumpSumCount 先转换为 Double 类型（避免整数除法截断），除以分母 pageSumCount，得到 0~1 之间的浮点数。

最后，`filter(col("jump").contains("->"))` 过滤掉最后一条记录产生的无跳转路径（不包含"->"的单个 pageid），`select(col("jump"), col("conversion"))` 仅保留跳转路径和转化率两列，`distinct()` 去重，`orderBy(col("jump"))` 按路径名排序。最终 result 包含 jump 和 conversion 两列。

3.5.4 写入 HBase conversion 表

与 3.4 节类似，本节也采用 foreachPartition 模式并行写入 HBase。程序首先在主线程中创建 HBaseUtil 实例，调用 `createTable("conversion")` 创建 conversion 表（列族 info），随后关闭该实例。然后调用 `result.rdd.foreachPartition`，每个分区独立创建 HBaseUtil 实例和 Connection，遍历分区内每行，将 jump 和 conversion 值封装为 conBean case class 实例（包含 jump 和 conv 两个字段），调用 `convBeanPut` 方法逐条写入 HBase。

conversion 表以 jump 路径字符串（如"page1->page2"）为 rowKey，info 列族下存储 conversion 列，列值为 Double 型的转化率。写入完成后各分区关闭各自的连接。

3.5.5 完整代码

Conversion.scala 的完整实现如下：

```scala
package example

import org.apache.spark.sql.expressions.{Window, WindowSpec}
import org.apache.spark.sql.{DataFrame, Dataset, Row, SparkSession}
import org.apache.spark.sql.functions._

object Conversion {
    def main(args: Array[String]): Unit = {
        val spark: SparkSession = SparkSession.builder().appName("Conversion").master("local[*]").getOrCreate()
        val sc = spark.sparkContext
        val dsdf: DataFrame = spark.read.json("C:\\data\\user_conversion.json")
        dsdf.show()

        val spec: WindowSpec = Window.partitionBy(col("userid")).orderBy(col("actionTime"))
        val conversionDF: DataFrame = dsdf.withColumn("index", row_number().over(spec))
            .withColumn("pageNext", lead(col("pageid"), 1).over(spec))
            .withColumn("jump", concat_ws("->", col("pageid"), col("pageNext")))

        val pageDF: DataFrame = conversionDF.groupBy(col("pageid")).agg(count("*").alias("pageSumCount"))
        val jumpDF: DataFrame = conversionDF.groupBy(col("jump")).agg(count("*").alias("jumpSumCount"))
        val result: Dataset[Row] = pageDF.join(conversionDF, Seq("pageid"), "inner").join(jumpDF, Seq("jump"), "inner")
            .withColumn("conversion", col("jumpSumCount").cast("Double") / col("pageSumCount"))
            .filter(col("jump").contains("->")).select(col("jump"), col("conversion"))
            .distinct().orderBy(col("jump"))

        val util = new HBaseUtil
        util.init()
        util.createTable("conversion")
        util.close()

        result.rdd.foreachPartition(item => {
            val util = new HBaseUtil
            util.init()
            item.foreach(row => {
                val jump: String = row.getAs[String]("jump")
                val conv: Double = row.getAs[Double]("conversion")
                val bean = new conBean(jump, conv)
                util.convBeanPut("conversion", bean)
            })
            util.close()
        })
    }
}
```

3.6 HBase 连接工具类封装（HBaseUtil.scala）

前三节的三项分析程序均依赖 HBase 进行结果持久化，为避免每个程序各自编写 HBase 连接与读写逻辑，系统将 HBase 操作统一封装为 HBaseUtil 工具类，供 Amazon、AmazonOrder 和 Conversion 三个分析程序共用。

3.6.1 数据模型定义

文件顶部定义了三个 case class 作为分析结果的数据载体：Info（id: String, view: Long, cart: Long, purchase: Long）对应品类行为分析的 Top10 结果；Product（addName: String, pro5115: Long, pro4856: Long, pro4767: Long）对应区域热门商品分析的三款商品统计结果；conBean（jump: String, conv: Double）对应页面单跳转化率分析的结果。Scala 的 case class 自动生成 apply、unapply、equals、hashCode 和 toString 方法，同时支持模式匹配，在 map 操作中可以简洁地将 DataFrame Row 映射为类型化对象。

3.6.2 连接管理与建表

HBaseUtil 类持有 `conn: Connection` 成员变量（初始值为 `_`，即未初始化），所有表操作共享同一连接实例。`init()` 方法通过 `HBaseConfiguration.create()` 创建配置对象，设置 ZooKeeper Quorum 地址为 `hadoop1:2181,hadoop2:2181,hadoop3:2181`，然后调用 `ConnectionFactory.createConnection(conf)` 建立连接。

`createTable(name: String)` 方法负责创建分析结果表。内部通过 `conn.getAdmin` 获取 Admin 管理接口，构造 TableDescriptorBuilder 并设置列族名为 info。若目标表已存在（重复运行分析程序时），先执行 `disableTable` 禁用再 `deleteTable` 删除旧表，确保每次写入的数据为最新结果。建表完成后关闭 Admin 实例。

3.6.3 批量写入方法

为适配三种分析结果的不同数据结构，HBaseUtil 提供了三个专用的写入方法：

- `infoPut(name: String, infoList: Seq[Info])`：接收 Info 序列，遍历构造 Put 实例。以 category_id 为 rowKey，在 info 列族下添加 view、cart、purchase 三列。所有 Put 收集为 Java List 后通过 `table.put(puts)` 一次性批量提交，利用 HBase 的批量写入机制减少 RPC 往返次数。

- `productPut(name: String, prod: Product)`：逐条写入单个 Product 对象，以 address_name 为 rowKey，写入 pro5115、pro4856、pro4767 三列。该方法配合 foreachPartition 在每个分区内逐行调用。

- `convBeanPut(name: String, bean: conBean)`：逐条写入单个 conBean 对象，以 jump 路径字符串为 rowKey，写入 conversion 列。

三个方法均使用 `Bytes.toBytes()` 进行类型转换：String 以 UTF-8 编码，Long 以 8 字节大端序编码，Double 以 8 字节 IEEE 754 双精度编码。写入完成后在 finally 块中关闭 Table 实例。

3.6.4 查询方法

HBaseUtil 还提供了两个查询方法供调试和验证使用。`infoGet(categoryId: String)` 根据 category_id 查询 categoryCount 表中的单行数据，读取 info:view、info:cart、info:purchase 三个列的值并封装为 Info 对象返回，若行不存在或任一列为 null 则返回 None。`productGet(name: String, addName: String)` 同理查询指定 address_name 对应的三款商品销量。两个方法均采用 `Option[T]` 返回类型，以函数式风格处理空值情况，避免 null 引用异常。

`close()` 方法关闭 Connection，释放与 ZooKeeper 和 RegionServer 的连接资源。各分析程序的 main 方法在完成写入后调用此方法。

3.6.5 完整代码

HBaseUtil.scala 的完整实现如下：

```scala
package example

import org.apache.hadoop.conf.Configuration
import org.apache.hadoop.hbase.HBaseConfiguration
import org.apache.hadoop.hbase.TableName
import org.apache.hadoop.hbase.client.{Admin, ColumnFamilyDescriptor, ColumnFamilyDescriptorBuilder, Connection, ConnectionFactory, Get, Put, Result, Table, TableDescriptor, TableDescriptorBuilder}
import org.apache.hadoop.hbase.util.Bytes

import scala.collection.JavaConverters._

case class Info(id: String, view: Long, cart: Long, purchase: Long)
case class Product(addName: String, pro5115: Long, pro4856: Long, pro4767: Long)
case class conBean(jump: String, conv: Double)

class HBaseUtil {
    var conn: Connection = _

    def init(): Unit = {
        val conf: Configuration = HBaseConfiguration.create()
        conf.set("hbase.zookeeper.quorum", "hadoop1:2181,hadoop2:2181,hadoop3:2181")
        conn = ConnectionFactory.createConnection(conf)
    }

    def createTable(name: String): Unit = {
        val admin: Admin = conn.getAdmin
        val tableName: TableName = TableName.valueOf(name)

        if (admin.tableExists(tableName)) {
            admin.disableTable(tableName)
            admin.deleteTable(tableName)
        }

        val tdb: TableDescriptorBuilder = TableDescriptorBuilder.newBuilder(tableName)
        val cfdb: ColumnFamilyDescriptorBuilder = ColumnFamilyDescriptorBuilder.newBuilder(Bytes.toBytes("info"))
        val cfd: ColumnFamilyDescriptor = cfdb.build()
        tdb.setColumnFamily(cfd)
        val td: TableDescriptor = tdb.build()
        admin.createTable(td)
        admin.close()
    }

    def infoPut(name: String, infoList: Seq[Info]): Unit = {
        if (infoList.isEmpty) return

        val table: Table = conn.getTable(TableName.valueOf(name))
        try {
            val cf: Array[Byte] = Bytes.toBytes("info")

            val puts: java.util.List[Put] = infoList.map { info =>
                val put: Put = new Put(Bytes.toBytes(info.id))
                put.addColumn(cf, Bytes.toBytes("view"), Bytes.toBytes(info.view))
                put.addColumn(cf, Bytes.toBytes("cart"), Bytes.toBytes(info.cart))
                put.addColumn(cf, Bytes.toBytes("purchase"), Bytes.toBytes(info.purchase))
                put
            }.asJava

            table.put(puts)
        } finally {
            table.close()
        }
    }

    def productPut(name: String, prod: Product): Unit = {
        val table: Table = conn.getTable(TableName.valueOf(name))
        try {
            val cf: Array[Byte] = Bytes.toBytes("info")
            val put: Put = new Put(Bytes.toBytes(prod.addName))
            put.addColumn(cf, Bytes.toBytes("pro5115"), Bytes.toBytes(prod.pro5115))
            put.addColumn(cf, Bytes.toBytes("pro4856"), Bytes.toBytes(prod.pro4856))
            put.addColumn(cf, Bytes.toBytes("pro4767"), Bytes.toBytes(prod.pro4767))
            table.put(put)
        } finally {
            table.close()
        }
    }

    def convBeanPut(name: String, bean: conBean): Unit = {
        val table: Table = conn.getTable(TableName.valueOf(name))
        try {
            val cf: Array[Byte] = Bytes.toBytes("info")
            val put: Put = new Put(Bytes.toBytes(bean.jump))
            put.addColumn(cf, Bytes.toBytes("conversion"), Bytes.toBytes(bean.conv))
            table.put(put)
        } finally {
            table.close()
        }
    }

    def productGet(name: String, addName: String): Option[Product] = {
        val table: Table = conn.getTable(TableName.valueOf(name))
        try {
            val get: Get = new Get(Bytes.toBytes(addName))
            get.addFamily(Bytes.toBytes("info"))
            val result: Result = table.get(get)

            if (result.isEmpty) {
                None
            } else {
                val cf: Array[Byte] = Bytes.toBytes("info")
                val pro5115Bytes: Array[Byte] = result.getValue(cf, Bytes.toBytes("pro5115"))
                val pro4856Bytes: Array[Byte] = result.getValue(cf, Bytes.toBytes("pro4856"))
                val pro4767Bytes: Array[Byte] = result.getValue(cf, Bytes.toBytes("pro4767"))

                if (pro5115Bytes == null || pro4856Bytes == null || pro4767Bytes == null) {
                    None
                } else {
                    Some(Product(addName, Bytes.toLong(pro5115Bytes), Bytes.toLong(pro4856Bytes), Bytes.toLong(pro4767Bytes)))
                }
            }
        } finally {
            table.close()
        }
    }

    def infoGet(categoryId: String): Option[Info] = {
        val table: Table = conn.getTable(TableName.valueOf("categoryCount"))
        try {
            val get: Get = new Get(Bytes.toBytes(categoryId))
            get.addFamily(Bytes.toBytes("info"))
            val result: Result = table.get(get)

            if (result.isEmpty) {
                None
            } else {
                val cf: Array[Byte] = Bytes.toBytes("info")
                val viewBytes: Array[Byte] = result.getValue(cf, Bytes.toBytes("view"))
                val cartBytes: Array[Byte] = result.getValue(cf, Bytes.toBytes("cart"))
                val purchaseBytes: Array[Byte] = result.getValue(cf, Bytes.toBytes("purchase"))

                if (viewBytes == null || cartBytes == null || purchaseBytes == null) {
                    None
                } else {
                    Some(Info(categoryId, Bytes.toLong(viewBytes), Bytes.toLong(cartBytes), Bytes.toLong(purchaseBytes)))
                }
            }
        } finally {
            table.close()
        }
    }

    def close(): Unit = {
        if (conn != null) {
            conn.close()
        }
    }
}
```

本章详细阐述了 session 项目的离线数据分析与处理过程。首先介绍了 Linux 集群的节点规划与软件版本配置，说明了 HBase 分布式集群的部署架构。随后描述了 user_session.json 和 user_conversion.json 两组数据源的字段结构与格式规范。

在此基础上，本章依次实现了三项 Spark SQL 分析程序：Amazon.scala 利用 DataFrame 的 filter、groupBy、agg 和 join 操作，按品类统计浏览、加购、购买三种行为次数并筛选 Top10，结果写入 HBase categoryCount 表；AmazonOrder.scala 运用 Window.partitionBy 与 row_number() 窗口函数，按州分组取各商品销量 Top3，提取三款特定商品数据后写入 HBase top3 表；Conversion.scala 借助 lead() 窗口函数计算用户页面跳转路径，通过除法运算得出每条跳转的转化率，结果写入 HBase conversion 表。最后介绍了 HBaseUtil 工具类的封装设计，包括三个 case class 数据模型、连接管理、建表、三种专用写入方法和查询方法的实现。

三项分析程序覆盖了 Spark SQL 中分组聚合、多表连接、窗口函数三种核心数据处理模式，形成了一套完整的离线数据分析链路。分析结果统一存储于 HBase 的三张表中，为下一章 Web 可视化系统的数据读取与展示提供了数据基础。

## 第四章 Web 可视化系统设计与实现（sparkView 项目）

4.1 系统总体架构设计

sparkView 是一个基于 Spring MVC 框架构建的 Web 可视化系统，部署于 Tomcat 9 容器，运行在 Windows 本地开发环境中。系统从 Linux 集群的 HBase 数据库中读取第三章的三项离线分析结果，通过后端 Java 代码进行数据格式转换，最终在前端以 ECharts 分组柱状图的形式呈现给用户。整个系统采用经典的分层架构，自上而下分为表示层、控制层、业务逻辑层、数据访问层和数据存储层五个层次。

表示层由 index.jsp 页面和 view.js 脚本文件组成，使用 ECharts 4.3.0 图表库和 jQuery 1.11.3 实现数据可视化与用户交互。用户点击左侧导航栏中的三个菜单项（十大火爆商品品类、五十州热销商品数据、网站页面单跳转化率），前端通过 jQuery 的 $.ajax 方法向对应后端接口发起 GET 请求，获取 JSON 数据后调用 echarts.setOption() 渲染分组柱状图。每个图表均配备 toolbox 工具栏，支持柱状图与折线图切换、数据视图查看以及图表保存为图片等交互功能。

控制层由 myController 类实现，标注 @Controller 注解，由 Servlet WebApplicationContext（springMVC.xml）扫描管理。myController 通过 @Autowired 注入业务层的 service 实例，定义了四个请求映射：根路径 "/" 返回 index.jsp 视图页面；"/top10"、"/top3"、"/jump" 三个路径均标注 @ResponseBody 注解，将返回的 Object[] 数组借助 Jackson 自动序列化为 JSON 格式响应。DispatcherServlet 作为前端控制器统一拦截所有请求，根据 HandlerMapping 定位到对应方法，方法执行完毕后由 HttpMessageConverter 完成 JSON 序列化输出。

业务逻辑层由 service 类实现，标注 @Service 注解，由 Root WebApplicationContext（ac.xml）扫描管理。service 通过 @Autowired 注入 DAO 层实例，提供 getTop10()、getTop3() 和 getConversion() 三个方法。每个方法的处理逻辑相同：调用 DAO 获取 ArrayList<Bean> 集合，遍历集合将 Bean 对象的各字段值拆分为独立的数组（如将 Top10Bean 列表拆分为 id[]、view[]、cart[]、purchase[] 四个数组），封装为 Object[] 返回。这种拆分是为了匹配 ECharts 分组柱状图所需的数据格式——xAxis 接收类别名称数组，series 中每个系列接收对应的数值数组。

数据访问层由 DAO 类和 Connect 类组成，分别标注 @Repository 和 @Component 注解。Connect 类封装了 HBase 的 ConnectionFactory 连接创建逻辑，提供 init() 方法返回 Connection 实例。DAO 类通过 @Autowired 注入 Connect 实例，提供 getTop10()、getTop3() 和 getConversion() 三个查询方法。每个方法均执行相同的操作流程：调用 Connect.init() 获取 HBase 连接，获取对应 Table 实例，创建 Scan 全表扫描器，遍历 ResultScanner 迭代器将每行 Result 的 Cell 数据逐一映射为对应的 Bean 对象（Top10Bean、Top3Bean 或 conversionBean），返回 ArrayList 集合。

数据存储层为部署于 Linux 集群的 HBase 数据库，包含第三章离线分析写入的三张结果表：categoryCount 表（rowKey 为 category_id，info 列族下存储 view/cart/purchase 三列），top3 表（rowKey 为 address_name，info 列族下存储 pro5115/pro4856/pro4767 三列），conversion 表（rowKey 为 jump 路径字符串，info 列族下存储 conversion 列）。sparkView 通过 ZooKeeper（hadoop1:2181,hadoop2:2181,hadoop3:2181）定位 HBase 集群并建立连接。

系统的整体架构如图 4-1 所示，各层之间的数据流向为自底向上的单向依赖：DAO 层通过 Connect 从 HBase 读取原始数据并封装为 Bean 对象，传递给 Service 层；Service 层将 Bean 集合拆分为符合 ECharts 格式要求的 Object[] 数组，返回给 Controller 层；Controller 层将数组通过 Jackson 序列化为 JSON，经 HTTP 响应发送至前端；前端 jQuery 的 AJAX 回调接收 JSON 后传入 ECharts 完成图表渲染。两级 Spring 上下文的分工也体现在图中：Root WebApplicationContext（ac.xml）管理数据访问层和业务逻辑层的组件，Servlet WebApplicationContext（springMVC.xml）管理控制层的组件，父子容器的依赖注入方向为子容器中的 Controller 可以引用父容器中的 Service，反之不可。
