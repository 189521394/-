# 全国大数据职位分析系统 —— 论文大纲

## 第一章 绪论

### 1.1 选题背景与意义

随着互联网的普及，网络招聘已成为求职与招聘的主要渠道。智联招聘、前程无忧等平台每天产生海量职位数据，涵盖岗位名称、薪资、学历、技能等丰富信息。但这些原始数据多为非结构化 JSON 格式，存在字段缺失、格式不统一等问题，无法直接用于统计分析。传统单机数据处理方式在面对大规模数据集时效率低下、扩展性差，而 Hadoop 分布式计算平台和 SSM 框架体系分别为离线数据处理和 Web 应用开发提供了成熟的解决方案。

基于此，本文设计并实现了一套全国大数据职位分析系统。系统采用三阶段架构：首先利用 MapReduce 对原始招聘数据进行清洗与格式转换，然后在 Hive 数据仓库中完成多维度聚合统计并通过 Sqoop 导出至 MySQL，最后基于 SSM 框架和 ECharts 可视化库构建 Web 端，以饼图、柱状图和词云图等形式直观展示全国计算机行业职位的分布格局。系统打通了从数据清洗到可视化展示的完整链路，对求职者择业和用人单位招聘具有一定的参考价值。

### 1.2 论文组织结构

本文共分六章。第一章介绍选题背景、研究现状和论文结构。第二章概述 Hadoop、Hive、Sqoop、SSM 框架、ECharts 等关键技术。第三章详细阐述数据来源、MapReduce 数据清洗实现、Hive 多维度分析以及 Sqoop 数据导出。第四章描述系统总体架构、功能模块划分和数据库设计。第五章具体说明 SSM 框架整合配置、三层架构实现及前端八个可视化图表的编码过程。第六章总结全文并展望改进方向。

## 第二章 相关技术介绍

### 2.1 Hadoop 分布式计算平台

Hadoop 是 Apache 基金会旗下的开源分布式计算平台，起源于 2003 年谷歌发表的 GFS、MapReduce 和 BigTable 三篇论文。Hadoop 的核心由 HDFS 分布式文件系统和 MapReduce 计算框架两部分组成，分别解决了海量数据的分布式存储与并行计算问题。它能够在由普通商用服务器组成的集群上运行，通过横向扩展节点数量来提升存储容量和计算能力，具备高可靠性、高扩展性和高容错性的特点。本系统利用 Hadoop 平台完成职位数据的离线清洗工作。

#### 2.1.1 HDFS 分布式文件系统

HDFS（Hadoop Distributed File System）是 Hadoop 的分布式文件系统，采用主从架构设计。集群中由单个 NameNode 负责管理文件系统的命名空间、维护文件与数据块的映射关系，多个 DataNode 负责实际存储数据块。HDFS 将大文件切分为固定大小的数据块（默认为 128MB），每个数据块默认保存三个副本，分布在不同的 DataNode 上，从而在节点故障时保证数据的可用性。HDFS 的设计理念是"一次写入、多次读取"，特别适合离线批量数据处理场景。本系统中，原始招聘数据集首先上传至 HDFS，作为 MapReduce 清洗任务的输入源。

#### 2.1.2 MapReduce 计算框架

MapReduce 是 Hadoop 的核心分布式计算模型，将复杂的并行计算抽象为两个阶段：Map（映射）阶段负责将输入数据分解为键值对并逐条处理，Reduce（归约）阶段负责将相同键的中间结果合并汇总。开发者只需实现 Map 和 Reduce 两个函数的具体逻辑，框架便自动完成任务调度、数据分发、负载均衡和容错处理等底层工作。MapReduce 的计算过程遵循"移动计算而非移动数据"的原则，将计算任务调度到数据所在的节点上执行，有效减少了网络传输开销。本系统通过 MapReduce 程序对原始招聘 JSON 文件进行逐条解析、字段清洗和格式转换，将处理结果写回 HDFS，为后续 Hive 数据分析提供干净的结构化数据。

### 2.2 Hive 数据仓库

Hive 是构建在 Hadoop 之上的数据仓库工具，由 Facebook 开源并贡献给 Apache 基金会。它定义了类 SQL 的查询语言 HQL（Hive Query Language），能够将结构化数据文件映射为数据库表，使用户无需编写复杂的 MapReduce 程序即可完成数据的查询与分析。Hive 在接收到 HQL 语句后，会自动将其翻译为 MapReduce 作业在集群上执行，底层数据存储在 HDFS 中。Hive 特别适合离线批量数据处理场景，但不适用于需要低延迟的实时查询。本系统在数据清洗完成后，将清洗结果加载至 Hive 中，利用 HQL 完成城市分布、学历要求、薪资水平等八个维度的聚合统计。

### 2.3 Sqoop 数据迁移工具

Sqoop（SQL to Hadoop）是 Apache 旗下的开源数据迁移工具，专门用于在 Hadoop 和关系型数据库之间高效传输数据。它可以将 MySQL、Oracle 等关系型数据库中的数据导入 HDFS 或 Hive，也可以将 Hadoop 中的处理结果导出到关系型数据库。Sqoop 的底层利用 MapReduce 实现数据的并行导入导出，具有良好的传输效率。本系统在 Hive 分析完成后，使用 Sqoop 将八张分析结果表从 HDFS 导出至 MySQL，为 Web 端的数据可视化提供查询支撑。

### 2.4 SSM 框架体系

SSM 是 Spring + Spring MVC + MyBatis 三个开源框架的整合体，是目前 Java Web 开发领域的主流技术栈。三者在分工上各有侧重：Spring 负责对象管理与依赖注入，Spring MVC 负责请求分发与表现层控制，MyBatis 负责数据库访问与对象映射。三者协同构成 Controller → Service → Mapper 的标准三层架构。本系统 Web 端采用 SSM 框架替代传统 Servlet + JDBC 的开发模式，大幅降低了代码耦合度，提升了开发效率和系统可维护性。

#### 2.4.1 Spring 框架与控制反转（IoC）

Spring 是一个轻量级的 Java 企业级开发框架，其核心是控制反转（Inversion of Control，IoC）容器。在传统开发模式中，对象之间的依赖关系由开发者手动创建和管理；在 Spring 中，对象的创建、装配和生命周期管理全部交由 IoC 容器负责，开发者只需通过注解（如 @Autowired）声明依赖关系，容器便自动完成注入。这种模式使各层之间的耦合度显著降低，便于代码测试与模块替换。本系统利用 Spring IoC 容器统一管理 Service 和 Mapper 实例，并通过 spring-tx 模块提供声明式事务支持。

#### 2.4.2 Spring MVC 表现层框架

Spring MVC 是构建在 Spring 之上的 Web 表现层框架，遵循模型-视图-控制器（MVC）设计模式。前端请求由 DispatcherServlet 统一接收，经 HandlerMapping 路由至对应的 Controller 处理方法，处理方法完成业务调用后返回模型数据，再由视图解析器渲染为 HTML 页面或直接输出 JSON 数据。Spring MVC 基于注解驱动开发，使用 @Controller 标识控制器类，@RequestMapping 映射请求路径，开发简便高效。本系统采用 Spring MVC 构建 RESTful 风格的数据接口，为前端 AJAX 请求提供 JSON 格式的响应数据。

#### 2.4.3 MyBatis 持久层框架

MyBatis 是一个半自动化的持久层框架，它封装了 JDBC 的底层操作，将 SQL 语句与 Java 代码解耦。与传统 JDBC 编程中手动建立连接、编写 SQL、遍历 ResultSet 再关闭连接的繁琐流程不同，MyBatis 支持通过注解（如 @Select、@Insert）或 XML 文件定义 SQL 映射，框架自动完成参数绑定和结果集到 Java 对象的映射。本系统使用 MyBatis 的注解式 Mapper 接口访问 MySQL 数据库，每个 Mapper 接口仅需一到两行注解代码即可完成数据查询，相比传统 JDBC 模式代码量减少约 90%。

#### 2.4.4 Jackson JSON 处理库

Jackson 是 Java 生态中广泛使用的 JSON 序列化与反序列化库。Spring MVC 内置了对 Jackson 的支持，当 Controller 方法标注 @ResponseBody 注解时，Spring MVC 自动调用 Jackson 将返回的 Java 对象序列化为 JSON 字符串写入 HTTP 响应体，前端 AJAX 请求即可直接获取结构化的 JSON 数据。在引入 Jackson 之前，开发者需要手动拼接 JSON 字符串，不仅代码冗长且容易出错，Jackson 的自动序列化机制极大简化了这一过程。本系统八个数据接口均通过 @ResponseBody + Jackson 实现 JSON 数据的自动输出。

### 2.5 MySQL 与 Druid 数据库连接池

MySQL 是目前应用最广泛的开源关系型数据库管理系统，具有体积小、速度快、成本低等特点，能够高效处理结构化数据的存储与查询。本系统使用 MySQL 作为最终分析结果的存储数据库，Hive 分析完成后的八张统计表通过 Sqoop 导出至 MySQL 的 job 数据库中。

Druid 是阿里巴巴开源的高性能数据库连接池组件。与传统 JDBC 编程中每次数据库操作都重复"建立连接—执行 SQL—关闭连接"的模式不同，Druid 在应用启动时预先创建一定数量的数据库连接并维护在连接池中，业务请求时直接从池中获取可用连接，使用完毕后归还而非销毁。这一机制显著降低了频繁创建和销毁连接的开销，同时 Druid 内置了 SQL 监控和防火墙功能，便于性能分析与安全防护。本系统通过 Spring 容器管理 Druid 数据源，在 ac.xml 中配置连接池参数，由 MyBatis 框架从池中获取连接执行 SQL。

### 2.6 ECharts 数据可视化库

ECharts 是百度开源的前端数据可视化图表库，基于 JavaScript 和 HTML5 Canvas 技术实现。它提供了折线图、柱状图、饼图、散点图、词云等数十种图表类型，支持图表的交互式操作（如缩放、拖拽、数据视图切换）和动态数据刷新。开发者只需传入 JSON 格式的数据并配置少量选项，即可渲染出高质量的图表。ECharts 兼容主流浏览器，具有良好的跨平台特性。本系统前端利用 ECharts 构建了八张可视化图表，包括三张饼图（城市分布、学历分布、经验要求）、两张柱状图（城市平均薪资、薪资区间分布）和三张词云图（岗位名称、技能需求、岗位福利），直观展示全国计算机行业职位的多维度分析结果。

### 2.7 其他工具与技术

#### 2.7.1 Maven 项目管理

Maven 是 Apache 旗下的 Java 项目构建与管理工具。通过一个名为 pom.xml 的配置文件，Maven 以声明式的方式管理项目的依赖库（如 Spring、MyBatis、Jackson 等），自动从中央仓库下载所需 JAR 包并处理依赖传递关系，无需开发者手动收集和管理第三方库。此外，Maven 提供了标准化的项目目录结构和构建生命周期（编译、测试、打包、部署），简化了项目从源码到可部署 WAR 包的完整流程。本系统使用 Maven 管理所有依赖，并执行 mvn clean package 命令完成项目构建。

#### 2.7.2 JSP / jQuery / AJAX

JSP（Java Server Pages）是 Java EE 规范中的动态页面技术，允许在 HTML 中嵌入 Java 代码，在服务器端渲染后生成标准 HTML 返回给浏览器。jQuery 是一个轻量级的 JavaScript 库，封装了 DOM 操作、事件处理和 AJAX 异步请求等常用功能，简化了前端开发。AJAX（Asynchronous JavaScript And XML）技术实现了浏览器与服务器之间的异步数据交互，使得页面无需整体刷新即可动态更新图表数据。本系统采用 JSP 作为视图层模板，前端通过 jQuery 封装 AJAX 请求调用后端数据接口，获取 JSON 数据后注入 ECharts 图表完成动态渲染。

## 第三章 数据采集与预处理

### 3.1 数据来源与说明

#### 3.1.1 数据集来源

鉴于主流招聘平台部署了高强度的反爬虫机制及严格的数据访问频率限制，且职位数据涉及企业内部招聘策略与隐私保护，为确保项目实施过程的合规性，本项目未采用实时全量爬取方案，而是采用了科研开源社区脱敏处理后的全国互联网行业招聘信息历史快照数据集。该数据集保留了完整的字段结构，包含岗位名称、薪资描述、学历要求、工作经验、城市、技能标签和福利列表等核心字段，数据量足以支撑后续分布式清洗与分析的功能验证。

#### 3.1.2 原始数据结构说明

原始数据集以 JSON 文件形式存储，每个文件包含多条职位记录。单条记录的核心字段如下：

| 字段 | 说明 | 示例值 |
|------|------|--------|
| encryptJobId | 加密岗位ID | "a1b2c3d4..." |
| jobName | 岗位名称 | "Java开发工程师" |
| salaryDesc | 薪资描述 | "15K-25K" 或 "15000-25000元/月" |
| jobDegree | 学历要求 | "本科" |
| jobExperience | 工作经验要求 | "3-5年" |
| cityName | 城市 | "北京" |
| skills | 技能标签（JSON数组） | ["Java","Spring","MySQL"] |
| welfareList | 福利列表（JSON数组） | ["五险一金","年终奖"] |

原始数据存在以下问题无法直接用于分析：薪资字段单位不统一（"K"与"元/月"混用）；字段中包含逗号、空格、连接符等特殊字符会干扰 Hive 的分隔符解析；部分字段存在空值或"N/A"标记。上述问题需要在数据清洗阶段统一处理。

### 3.2 数据预处理总体方案

#### 3.2.1 数据预处理技术概述

现实世界中的数据大多是不完整、不一致的"脏数据"，无法直接用于数据挖掘或统计分析。数据预处理正是为了解决这一问题而提出的技术手段，主要包括以下方法：

**数据清洗**是指检测并纠正数据集中存在的错误和不一致，包括处理缺失值、修正格式错误、剔除重复记录和统一编码格式等操作，目的是提升数据质量使其满足分析要求。

**数据集成**是将来自多个数据源的异构数据合并为一个统一数据集合的过程，需要解决不同源之间的字段命名冲突、度量单位不一致和数值重复等问题。

**数据变换**是将数据转换或统一为适合分析的表示形式，包括数据标准化、连续值离散化、类型转换（如字符串转数值）和单位换算等操作。

本系统的预处理工作同时涉及以上三个方面：使用 MapReduce 程序解析 JSON 并清洗不规范字段（数据清洗），将分散的多个 JSON 文件合并处理为统一格式的输出（数据集成），以及对薪资字段进行单位统一转换（数据变换）。

#### 3.2.2 本系统预处理流程设计

本系统的数据预处理流程分为四个阶段，以下为各阶段说明：

**第一阶段：数据准备。** 将科研开源数据集上传至 Hadoop 集群，在 HDFS 中创建 /job/in 目录作为 MapReduce 作业的输入路径。

**第二阶段：MapReduce 数据清洗。** 这是预处理的核心环节。通过自行编写的 MapReduce 程序对原始 JSON 文件进行逐条解析，完成字段提取、特殊字符替换、薪资单位统一和空值处理等清洗操作，输出以 \001 为分隔符的结构化文本文件到 HDFS 的 /job/out 目录。

**第三阶段：Hive 数据分析。** 在 Hive 中创建外部表加载清洗后的数据，执行数据去重后，利用 HQL 语句对城市、学历、经验、岗位名称、技能、福利、薪资七个维度进行聚合统计，生成八张分析结果表。

**第四阶段：Sqoop 数据导出。** 在 MySQL 中预先创建对应的八张目标表，通过 Sqoop 将 Hive 中的分析结果导出至 MySQL，完成从大数据处理平台到关系型数据库的数据迁移，为 Web 端提供快速的 SQL 查询支撑。

### 3.3 MapReduce 数据清洗实现

数据清洗是预处理流程的核心环节。本节通过 MapReduce 框架编写六个 Java 类，依次完成 JSON 解析、字段提取、特殊字符替换、薪资单位统一等清洗操作，最终输出以 `\001` 为分隔符的结构化文本文件。

#### 3.3.2 JobClean 数据清洗类

JobClean 类负责逐条解析 JSON 数据，提取各字段并完成特殊字符替换和薪资单位统一转换，最终以 `\001` 分隔符拼接输出：

```java
package position;

import org.codehaus.jettison.json.JSONArray;
import org.codehaus.jettison.json.JSONException;
import org.codehaus.jettison.json.JSONObject;

public class JobClean {
    public static String cleanJob(JSONArray jobList) throws JSONException {
        String jobCleanData = "";
        for (int i = 0; i < jobList.length(); i++) {
            JSONObject jsonListObject = jobList.getJSONObject(i);

            String jobId = exField(jsonListObject, "encryptJobId");
            String jobIdNew = parseString(jobId);

            String jobName = exField(jsonListObject, "jobName");
            String jobNameNew = parseString(jobName);

            String salaryDesc = exField(jsonListObject, "salaryDesc");
            String salaryNew = salaryResult(salaryDesc);

            JSONArray skills = jsonListObject.optJSONArray("skills");
            String skillNew = parseJSON(skills);

            String jobDegree = exField(jsonListObject, "jobDegree");
            String jobDegreeNew = parseString(jobDegree);

            String cityName = exField(jsonListObject, "cityName");
            String cityNameNew = parseString(cityName);

            JSONArray welfareList = jsonListObject.optJSONArray("welfareList");
            String welfareListNew = parseJSON(welfareList);

            String jobExperience = jsonListObject.getString("jobExperience");
            String jobExperienceNew = parseString(jobExperience);

            if (i == jsonListObject.length()-1) {
                jobCleanData = jobCleanData + jobIdNew + "\001" + jobNameNew + "\001" + salaryNew + "\001" + jobExperienceNew + "\001" + skillNew + "\001" + jobDegreeNew + "\001" + cityNameNew + "\001" + welfareListNew;
            } else {
                jobCleanData = jobCleanData + jobIdNew + "\001" + jobNameNew + "\001" + salaryNew + "\001" + jobExperienceNew + "\001" + skillNew + "\001" + jobDegreeNew + "\001" + cityNameNew + "\001" + welfareListNew + "\n";
            }
        }
        return jobCleanData;
    }

    public static String parseJSON(JSONArray skills) throws JSONException {
        if (skills == null || skills.length() == 0) {
            return null;
        }
        String skillName = "";
        for (int i = 0; i < skills.length(); i++) {
            skillName += skills.getString(i) + "-";
        }
        if (!skillName.isEmpty()) {
            return skillName.substring(0, skillName.length() - 1);
        } else {
            return null;
        }
    }

    public static String salaryResult(String salary) {
        if (salary.isEmpty()) {
            return "";
        }
        StringBuffer buffer = new StringBuffer("");

        if (salary.contains("K")) {
            return salary.substring(0, salary.indexOf("K"));
        } else if (salary.contains("元")) {
            String s = salary.substring(0, salary.indexOf("元"));
            String[] split = s.split("-");
            for (String temp : split) {
                int i = Integer.parseInt(temp);
                buffer.append(i * 30 / 1000 + "-");
            }
            return buffer.toString().substring(0, buffer.length() - 1);
        } else {
            return "0-0";
        }
    }

    public static String parseString(String string) {
        if (string == null || string.equals("N/A")) {
            return "N/A";
        }
        return string.replace(",", "/").replace("，", "/").replace(" ", "/").replace("-", "/");
    }

    public static String exField(JSONObject job,String fieldName) throws JSONException {
        if (job.has(fieldName) && !job.isNull(fieldName)) {
            return job.getString(fieldName);
        }
        return "N/A";
    }
}
```

核心方法说明：`cleanJob()` 遍历 JSON 数组调用各辅助方法逐字段清洗；`salaryResult()` 统一薪资单位（将"元/月"转换为"K"）；`parseString()` 替换逗号、空格、连接符等会干扰 Hive 分隔符解析的特殊字符；`parseJSON()` 将 JSON 数组以 `-` 分隔拼接为字符串。

#### 3.3.3 WholeFileRecordReader 自定义 RecordReader

由于原始数据集中存在大量小 JSON 文件，默认的 TextInputFormat 会逐行读取破坏 JSON 结构。WholeFileRecordReader 继承自 RecordReader，将每个文件的内容作为一个整体读取：

```java
package position;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.FSDataInputStream;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.BytesWritable;
import org.apache.hadoop.io.IOUtils;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.InputSplit;
import org.apache.hadoop.mapreduce.RecordReader;
import org.apache.hadoop.mapreduce.TaskAttemptContext;
import org.apache.hadoop.mapreduce.lib.input.FileSplit;

import java.io.IOException;

public class WholeFileRecordReader extends RecordReader<Text, BytesWritable> {
    private FileSplit fileSplit;
    private Configuration conf;
    private boolean processed = false;
    private Text key = new Text();
    private BytesWritable value = new BytesWritable();

    @Override
    public void initialize(InputSplit Split, TaskAttemptContext Context) throws IOException, InterruptedException {
        this.fileSplit = (FileSplit) Split;
        this.conf = Context.getConfiguration();
        this.processed = false;
    }

    @Override
    public boolean nextKeyValue() throws IOException, InterruptedException {
        if (!processed) {
            Path file = fileSplit.getPath();
            FileSystem fs = file.getFileSystem(conf);
            FSDataInputStream in = fs.open(file);

            byte[] contents = new byte[(int) fileSplit.getLength()];
            in.readFully(contents);

            key.set(file.getName());
            value.set(contents, 0, contents.length);

            IOUtils.closeStream(in);
            processed = true;
            return true;
        }
        return false;
    }

    @Override
    public Text getCurrentKey() throws IOException, InterruptedException {
        return key;
    }

    @Override
    public BytesWritable getCurrentValue() throws IOException, InterruptedException {
        return value;
    }

    @Override
    public float getProgress() throws IOException, InterruptedException {
        return processed ? 1.0F : 0.0F;
    }

    @Override
    public void close() throws IOException {

    }
}
```

#### 3.3.4 WholeFileInputFormat 自定义 FileInputFormat

WholeFileInputFormat 继承自 FileInputFormat，将 `isSplitable()` 设为 `false` 禁止文件分片，保证每个 JSON 文件作为一个完整记录处理：

```java
package position;

import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.BytesWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.InputSplit;
import org.apache.hadoop.mapreduce.JobContext;
import org.apache.hadoop.mapreduce.RecordReader;
import org.apache.hadoop.mapreduce.TaskAttemptContext;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;

import java.io.IOException;

public class WholeFileInputFormat extends FileInputFormat<Text, BytesWritable> {
    @Override
    protected boolean isSplitable(JobContext context, Path filename) {
        return false;
    }

    @Override
    public RecordReader<Text, BytesWritable> createRecordReader(InputSplit Split, TaskAttemptContext Context) throws IOException, InterruptedException {
        WholeFileRecordReader reader = new WholeFileRecordReader();
        reader.initialize(Split,Context);
        return reader;
    }
}
```

#### 3.3.5 JobMapper 映射类

JobMapper 在 Map 阶段接收文件内容，将其解析为 JSON 对象后调用 JobClean.cleanJob() 进行清洗，输出清洗后的结构化文本：

```java
package position;

import org.apache.hadoop.io.BytesWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;
import org.codehaus.jettison.json.JSONArray;
import org.codehaus.jettison.json.JSONException;
import org.codehaus.jettison.json.JSONObject;

import java.io.IOException;

public class JobMapper extends Mapper<Text, BytesWritable, Text, NullWritable> {
    @Override
    protected void map(Text key, BytesWritable value, Mapper<Text, BytesWritable, Text, NullWritable>.Context context) throws IOException, InterruptedException {
        String fileName = key.toString();
        String fileContent = new String(value.getBytes(), 0, value.getLength());
        try {
            String positionData = fileContent.replace(" ", "");
            JSONObject positionObject = new JSONObject(positionData);
            JSONObject zpData = positionObject.getJSONObject("zpData");
            JSONArray jobList = zpData.getJSONArray("jobList");
            String jobCleanData = JobClean.cleanJob(jobList);
            context.write(new Text(jobCleanData), NullWritable.get());
        } catch (JSONException e) {
            throw new RuntimeException(e);
        }
    }
}
```

#### 3.4.4 薪资数据处理

薪资字段的处理较为复杂，需要依次完成范围拆分、类型转换、平均值计算和区间分布统计四个步骤：

```sql
-- 1）拆分薪资上下限
create table salary_tmp1 as
select city,
substring(salary,0,instr(salary,'-')-1) as min_salary,
substring(salary,instr(salary,'-')+1) as max_salary
from job_ods_new;

-- 2）转换为数值类型并计算岗位平均薪资
create table salary_tmp2 as
select city,
(cast(min_salary as int) + cast(max_salary as int)) / 2 as salary_avg
from salary_tmp1;

-- 3）计算各城市平均薪资
create table salary_avg as
select city, round(avg(salary_avg),2) as savg
from salary_tmp2
group by city
order by savg DESC;

-- 4）按薪资范围拆分薪资区间
-- 预先创建表
create table salary_count_tmp (
    sort_key int,
    salary_range string
)
row format delimited
fields terminated by '\001';

-- 插入数据（过滤 NULL 值，按5k区间分组）
insert into table salary_count_tmp
select
    cast(floor(salary_avg / 5) * 5 as int),
    concat(
        cast(floor(salary_avg / 5) * 5 as int),
        '-',
        cast((floor(salary_avg / 5) + 1) * 5 as int),
        'k'
    )
from salary_tmp2
where salary_avg is not null;

-- 5）合并薪资区间，得到最终薪资范围表
create table salary_count as
select
    sort_key,
    salary_range,
    count(*) as num
from salary_count_tmp
group by sort_key, salary_range
order by sort_key asc;
```

## 第四章 系统设计

### 4.1 系统总体架构

本系统采用"数据采集→离线分析→Web 可视化"的三阶段分层架构设计。各层之间通过 HDFS 文件读写、Sqoop JDBC 传输和 HTTP JSON 响应三种机制实现数据流转，自下而上逐步将原始非结构化招聘数据转化为直观的可交互图表。系统总体架构如图 4-1 所示。

![系统总体架构图](系统总体架构图.png)

**图 4-1 系统总体架构**

**数据采集与预处理层**位于架构最顶端，负责将原始 JSON 数据集接入 HDFS 后，通过自行编写的 MapReduce 程序（JobDriver 驱动、JobMapper 映射、JobClean 清洗，配合 WholeFileInputFormat 和 WholeFileRecordReader 实现小文件整体读取）完成 JSON 解析、特殊字符替换和薪资单位统一，清洗后的结构化文本落地至 HDFS 供下一层消费。

**数据分析与导出层**承接清洗结果：首先在 Hive 中创建外部表并加载数据，经过去重后利用 HQL 完成城市分布、学历要求、工作经验、岗位名称、技能需求、岗位福利、城市平均薪资和薪资区间分布共八个维度的聚合统计，生成八张分析结果表；随后通过 Sqoop 将这八张表批量导出至 MySQL 关系型数据库，实现从分布式文件系统到关系型数据库的数据迁移，为 Web 端提供低延迟的 SQL 查询支撑。

**Web 可视化层**基于 SSM 框架构建：Spring IoC 容器统一管理 Service 和 Mapper 实例，Spring MVC 提供 8 个 RESTful 风格的 JSON 数据接口，MyBatis 注解式 Mapper 以极简代码完成数据库查询；前端通过 jQuery AJAX 异步请求各接口获取 JSON 数据，注入 ECharts 渲染为饼图、柱状图和词云图，最终在浏览器中呈现可交互的数据可视化仪表板。

### 4.2 功能模块设计

根据系统总体架构，本系统划分为数据清洗、数据分析、数据导出和 Web 可视化四个功能模块，各模块职责明确、边界清晰。系统功能模块如图 4-2 所示。

![系统功能模块图](功能模块图.png)

**图 4-2 系统功能模块图**

**数据清洗模块**部署于 Hadoop 集群，是数据处理链路的入口模块。该模块包含四个子功能：JSON 解析与字段提取（从原始 JSON 结构中定位并抽取 encryptJobId、jobName、salaryDesc 等 8 个核心字段）、特殊字符替换（将字段值中的逗号、空格、连接符等统一替换为"/"，避免干扰后续 Hive 的分隔符解析）、薪资单位统一（识别"K"和"元/月"两种格式，将"元/月"统一换算为"K"）、小文件合并读取（通过自定义 FileInputFormat 和 RecordReader 将多个小 JSON 文件按完整文件粒度读入 Map 任务）。

**数据分析模块**在 Hive 数据仓库中运行，是数据处理链路的计算核心。该模块包含三个子功能：基础维度统计（对学历、城市、工作经验和岗位名称四个维度分别执行 GROUP BY 聚合，生成对应的统计表）、数组字段拆分统计（使用 explode() 函数将技能标签数组和福利列表数组拆分为单行后再进行词频统计）、薪资数据处理（依次完成薪资范围拆分、字符串到数值的类型转换、城市平均薪资计算和以 5K 为步长的区间分布统计）。

**数据导出模块**是连接大数据处理平台与 Web 应用端的桥梁。该模块首先在 MySQL 中创建与 Hive 分析结果表字段一致的八张目标表，随后通过 Sqoop export 命令将八张表从 HDFS 并行导出至 MySQL，每条命令指定 JDBC 连接信息、目标表名、字段映射和 HDFS 源路径。

**Web 可视化模块**是系统与用户的交互界面。该模块以 SSM 三层架构为基础：Mapper 层通过 @Select 注解定义 8 条 SQL 查询；Service 层封装业务逻辑并组装返回数据；Controller 层提供 `/city`、`/degree`、`/experience`、`/avgSalary`、`/salary`、`/jobName`、`/skills`、`/welfare` 共 8 个 GET 接口，以 @ResponseBody + Jackson 自动序列化返回 JSON。前端 index.jsp 页面采用侧边栏导航布局，通过 jQuery AJAX 调用上述接口并动态渲染三张饼图（城市分布、学历分布、经验分布）、两张柱状图（城市平均薪资、薪资区间分布）和三张词云图（岗位名称、技能需求、岗位福利）。

### 4.3 数据库设计

本系统 MySQL 数据库名为 job，存储 Sqoop 从 Hive 导出的八张分析结果表。由于这些表是独立维度统计的最终产物，表之间不存在外键关联关系，因此本节直接以表格形式给出各表的结构定义。每张表仅包含维度名称和统计数量两个核心字段（薪资相关表额外包含排序键或薪资范围字段），结构简洁、查询高效。

### 5.2 SSM 框架整合配置

SSM 框架整合是 Web 端工程化的基础，核心在于搭建两级 Spring 上下文（父子容器）架构：Root WebApplicationContext 管理 Druid 数据源、SqlSessionFactory、Mapper 和 Service 等基础组件；Servlet WebApplicationContext 仅管理 Controller 层。以下依次说明四个配置文件的编写。

#### 5.2.1 Spring 与 MyBatis 整合（ac.xml）

`ac.xml` 是 Root 容器的配置文件，由 ContextLoaderListener 在启动时加载。它完成三项任务：加载 `jdbc.properties` 并创建 Druid 数据源、创建 SqlSessionFactory、扫描 Mapper 和 Service 组件：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xsi:schemaLocation="http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd http://www.springframework.org/schema/context https://www.springframework.org/schema/context/spring-context.xsd">
    <context:component-scan base-package="org.service"></context:component-scan>
    <context:property-placeholder location="classpath:jdbc.properties"></context:property-placeholder>
    <bean id="dataSource" class="com.alibaba.druid.pool.DruidDataSource">
        <property name="driverClassName" value="${jdbc.driverClass}"></property>
        <property name="url" value="${jdbc.url}"></property>
        <property name="username" value="${jdbc.username}"></property>
        <property name="password" value="${jdbc.password}"></property>
    </bean>
    <bean id="sqlSessionFactory" class="org.mybatis.spring.SqlSessionFactoryBean">
        <property name="dataSource" ref="dataSource"></property>
    </bean>
    <bean class="org.mybatis.spring.mapper.MapperScannerConfigurer">
        <property name="basePackage" value="org.mapper"></property>
    </bean>
</beans>
```

```xml
    <context:component-scan base-package="org.controller"></context:component-scan>
    <mvc:annotation-driven></mvc:annotation-driven>

```

#### 5.2.3 Web 部署描述符配置（web.xml）

`web.xml` 定义 ContextLoaderListener（加载 `ac.xml`）和 DispatcherServlet（加载 `springMVC.xml`）的初始化顺序，形成两级父子容器：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="http://xmlns.jcp.org/xml/ns/javaee"

         version="4.0">
    <context-param>
        <param-name>contextConfigLocation</param-name>
        <param-value>classpath:ac.xml</param-value>
    </context-param>
    <listener>
        <listener-class>org.springframework.web.context.ContextLoaderListener</listener-class>
    </listener>
    <servlet>
        <servlet-name>DispatcherServlet</servlet-name>
        <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
        <init-param>
            <param-name>contextConfigLocation</param-name>
            <param-value>classpath:springMVC.xml</param-value>
        </init-param>
        <load-on-startup>1</load-on-startup>
    </servlet>
    <servlet-mapping>
        <servlet-name>DispatcherServlet</servlet-name>
        <url-pattern>/</url-pattern>
    </servlet-mapping>
</web-app>
```

### 5.4 业务逻辑层实现

Service 层位于 Controller 和 Mapper 之间，负责调用 Mapper 获取原始结果并根据前端图表格式需求进行数据转换。系统定义 8 个 Service 类，以 `@Service` 标注并 `@Autowired` 注入对应 Mapper。以下列出全部 8 个 Service：

CityService — 城市分布：

```java
package org.service;

import org.bean.CityCountBean;
import org.bean.DataBean;
import org.mapper.CityMapper;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.ArrayList;

@Service
public class CityService {
    @Autowired
    private CityMapper cityMapper;

    public DataBean[] getCityCountAll() {
        ArrayList<CityCountBean> cityCountAll = cityMapper.getCityCountAll();
        DataBean[] dataBeans = new DataBean[cityCountAll.size()];
        for (int i = 0; i < cityCountAll.size(); i++) {
            CityCountBean cityCountBean = cityCountAll.get(i);
            DataBean dataBean = new DataBean(cityCountBean.getCity(), cityCountBean.getCity_count() + "");
            dataBeans[i] = dataBean;
        }
        return dataBeans;
    }
}
```

```java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.ArrayList;

@Service
public class SalaryService {

    private SalaryMapper salaryMapper;

        ArrayList<SalaryBean> salaryAll = salaryMapper.getSalaryAll();
        Object[] objects = new Object[2];
        String[] salary_range = new String[salaryAll.size()];
        Integer[] salary_count = new Integer[salaryAll.size()];
        for (int i = 0; i < salaryAll.size(); i++) {
```

### 5.5 控制层实现

Controller 是 Spring MVC 的请求处理层，统一接收前端 AJAX 请求、调用 Service 并返回 JSON 数据。系统采用单一 Controller 模式，所有 9 个路由集中定义在 `myController` 类中。

#### 5.5.1 Controller 路由设计

`myController` 标注 `@Controller`，通过 `@Autowired` 注入 8 个 Service 实例。类内定义 9 个方法，`@RequestMapping` 映射 URL 路径，`@ResponseBody` 实现 JSON 自动序列化。其中 `/` 返回 JSP 视图页面，其余 8 个路径为数据接口：

```java
package org.controller;

import org.bean.DataBean;
import org.service.*;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.ResponseBody;
import org.springframework.web.servlet.ModelAndView;

@Controller
public class myController {

    @Autowired
    private CityService cityService;
    @Autowired
    private DegreeService degreeService;
    @Autowired
    private ExpService expService;
    @Autowired
    private SalaryAvgService salaryAvgService;
    @Autowired
    private SalaryService salaryService;
    @Autowired
    private JobNameService jobNameService;
    @Autowired
    private SkillService skillService;
    @Autowired
    private WelfareService welfareService;

    @RequestMapping("/")
    public ModelAndView getBook() {
        ModelAndView modelAndView = new ModelAndView();
        modelAndView.setViewName("/WEB-INF/views/index.jsp");
        return modelAndView;
    }

    @RequestMapping("/city")
    @ResponseBody
    public DataBean[] getCity() {
        return cityService.getCityCountAll();
    }

    @RequestMapping("/degree")
    @ResponseBody
    public DataBean[] getDegree() {
        return degreeService.getDegreeCountAll();
    }

    @RequestMapping("/experience")
    @ResponseBody
    public DataBean[] getExperience() {
        return expService.getExpCountAll();
    }

    @RequestMapping("/avgSalary")
    @ResponseBody
    public Object[] getAvgSalary() {
        return salaryAvgService.getSalaryAvgAll();
    }

    @RequestMapping("/salary")
    @ResponseBody
    public Object[] getSalary() {
        return salaryService.getSalaryRangeAll();
    }

    @RequestMapping("/jobName")
    @ResponseBody
    public DataBean[] getJobName() {
        return jobNameService.getJobNameAll();
    }

    @RequestMapping("/skills")
    @ResponseBody
    public DataBean[] getSkills() {
        return skillService.getSkillCountAll();
    }

    @RequestMapping("/welfare")
    @ResponseBody
    public DataBean[] getWelfare() {
        return welfareService.getWelfareCountAll();
    }
}
```

Controller 方法的实现高度精简，每个业务路由方法仅一行代码——调用 Service 方法并直接返回结果。@ResponseBody 注解使 Spring MVC 自动调用 Jackson 将返回值序列化为 JSON 字符串写入 HTTP 响应体。

#### 5.5.2 @ResponseBody 与 Jackson 自动 JSON 序列化

当 Controller 方法标注 `@ResponseBody` 时，Spring MVC 跳过视图解析流程，直接调用 `MappingJackson2HttpMessageConverter` 将返回的 Java 对象序列化为 JSON 字符串写入 HTTP 响应体，Content-Type 自动设为 `application/json`。前端 jQuery AJAX 收到响应后自动将 JSON 反序列化为 JavaScript 对象，success 回调中的 data 参数可直接使用，无需手动调用 `JSON.parse()`。

#### 5.5.3 两级 Spring 上下文（父子容器）机制

本系统采用两级 Spring 上下文架构。Tomcat 启动时，`ContextLoaderListener` 首先加载 `ac.xml` 创建 Root WebApplicationContext，完成 Druid 数据源、SqlSessionFactory、Mapper 和 Service 的初始化。随后 `DispatcherServlet` 加载 `springMVC.xml` 创建 Servlet WebApplicationContext，将 Root 容器设为父容器。Controller 注入 Service 时，Spring 先在 Servlet 容器中查找，未找到则向父容器（Root）查找，形成单向依赖关系。这种架构使数据源等基础设施与 Controller 层完全解耦，各层职责清晰隔离。

### 注释

**1. 关于数据集来源与合规性的说明**

本系统所使用的原始招聘数据集来源于科研开源社区脱敏处理后的历史快照，并非通过爬虫程序从主流招聘平台实时采集。采用该数据集的主要原因有二：一是主流招聘平台部署了高强度的反爬虫机制及严格的访问频率限制，批量采集不可行；二是职位数据涉及企业内部招聘策略与个人信息，脱敏处理后的开源数据规避了隐私合规风险。该数据集保留了完整的字段结构，足以支撑 MapReduce 清洗和 Hive 分析的完整功能验证。在实际产业环境中，数据源可替换为经合法授权的实时职位数据。

**2. 关于 Hadoop 运行模式的说明**

本系统的 MapReduce 清洗作业在单节点 Hadoop 伪分布式环境下运行。考虑到原始数据集规模有限（与生产环境中 TB 级日志数据不同），单节点配置已足以完成功能验证。若需体现 Hadoop 的分布式计算优势，可将清洗作业提交至多节点集群执行。Hive 和 Sqoop 同样部署于同一节点，通过 `hive_local` 模式运行，数据存储在本地文件系统而非 HDFS 多副本分布式存储中。

**3. 关于中间临时表的说明**

第三章 Hive 分析过程中产生了多张中间临时表（如 `salary_tmp1`、`salary_tmp2`、`skill_tmp`、`welfare_tmp` 等），这些表仅服务于后续计算步骤，分析完成后可删除以释放存储空间。保留它们是为了便于读者追踪薪资三段式处理（拆分→转换→聚合）和数组字段拆分统计的完整流程。在生产环境中可将中间步骤合并为嵌套子查询，减少临时表创建。

**4. 关于 SSM 架构中 WEB-INF 隔离的说明**

本系统前端页面 `index.jsp` 放置在 `/WEB-INF/views/` 目录下，用户无法通过浏览器直接输入 URL 访问该页面，必须经由 Spring MVC 的 Controller 路由转发。这是 Java Web 开发中的安全实践，避免了直接暴露 JSP 视图路径带来的安全隐患。
