## 二、技术栈对比

### 2.1 整体架构

| 层次 | 老项目（大一） | 新项目（jobSSM） | 变更说明 |
|------|---------------|-------------------|----------|
| **构建工具** | 无（手动管理 jar 包） | Maven（pom.xml） | 引入依赖管理，版本可控 |
| **后端框架** | 原生 Servlet（无框架） | **Spring 5.2.8 + Spring MVC 5.2.8 + MyBatis 3.5.2** | 从零框架到标准 SSM |
| **数据库连接** | 手动 JDBC（DriverManager） | **Druid 1.1.20 连接池**（Spring 托管） | 连接池提升性能与稳定性 |

| **事务管理** | 无 | **spring-tx 5.2.8** | 声明式事务支持 |
| **配置方式** | 硬编码在 Java 类中（数据库 URL、用户名、密码写死在代码里） | 外部化配置文件（jdbc.properties + ac.xml + springMVC.xml） | 配置与代码分离 |

### 3.2 数据访问层对比

**老项目（大一）** — 手动 JDBC：
```java
// 数据库连接信息硬编码在代码中
Class.forName("com.mysql.cj.jdbc.Driver");
Connection cn = DriverManager.getConnection(
    "jdbc:mysql://localhost:3306/job?...", "root", "114514");

// 手动编写 SQL + ResultSet 遍历映射
Statement stmt = cn.createStatement();
ResultSet rs = stmt.executeQuery("select * from city_count");
while (rs.next()) {
    CityCountBean bean = new CityCountBean();
    bean.setCity(rs.getString("city"));
    bean.setCity_count(rs.getInt("city_count"));
    list.add(bean);
}
// 手动关闭连接
cn.close();
```

**新项目（jobSSM）** — MyBatis 注解式 Mapper：
```java
@Mapper
public interface CityMapper {
    @Select("select * from city_count order by city_count desc")
    public ArrayList<CityCountBean> getCityCountAll();
}
```

> **关键改进**：
> - 连接管理：`DriverManager` 手动连接 → Druid 连接池（Spring 托管，自动回收）
> - SQL 执行：手动 `Statement/ResultSet` → MyBatis `@Select` 注解，自动映射到 Bean
> - 每张表代码量：~50 行 → ~5 行（减少 90%）
> - 无需手动 try-catch-finally 关闭连接

### 3.3 业务层对比

**老项目（大一）**：手动 new DAO 对象，无事务管理

**新项目（jobSSM）**：
```java
@Service
public class CityService {
    @Autowired
    private CityMapper cityMapper;   // Spring IoC 自动注入

    public DataBean[] getCityCountAll() {
        ArrayList<CityCountBean> list = cityMapper.getCityCountAll();
        DataBean[] result = new DataBean[list.size()];
        for (int i = 0; i < list.size(); i++) {
            result[i] = new DataBean(list.get(i).getCity(),
                                     list.get(i).getCity_count() + "");
        }
        return result;
    }
}
```

> **关键改进**：
> - `@Service` + `@Autowired` 替代手动 new，由 Spring 容器管理 Bean 生命周期
> - 业务逻辑清晰：Mapper 取数据 → 转统一 DataBean → 返回

### 3.4 控制层对比

**老项目（大一）**：原始 Servlet，手动配置 web.xml 路由，手动拼接 JSON 字符串

**新项目（jobSSM）**：
```java
@Controller
public class myController {
    @Autowired
    private CityService cityService;
    // ... 8 个 Service 注入

    @RequestMapping("/")
    public ModelAndView getBook() { ... }      // 首页路由

    @RequestMapping("/city")
    @ResponseBody
    public DataBean[] getCity() {              // REST JSON 接口
        return cityService.getCityCountAll();
    }
    // ... 共 8 个数据接口
}
```

> **关键改进**：
> - `@Controller` + `@RequestMapping` 声明式路由，无需配置 XML
> - `@ResponseBody` + Jackson 自动将返回值序列化为 JSON
> - 单一 Controller 集中管理所有接口
> - Spring IoC 统一注入 8 个 Service

## 四、数据流对比

### 老项目数据流

```
原始招聘数据 → MapReduce 清洗（自己编写） → Hive 分析 → Sqoop → MySQL
  ↓
手动 JDBC（MyCn.java）
  ↓
DAO 层（手动 SQL + ResultSet 遍历）
  ↓
Service 层（手动 new 对象）
  ↓
Servlet（手动拼 JSON 字符串）
  ↓
JSP + jQuery + ECharts（简单展示）
```

### 新项目数据流

```
MySQL（8 张清洗好的分析结果表，数据已有）
  ↓
Druid 连接池（Spring 托管）
  ↓
MyBatis Mapper（@Select 注解 + 自动映射）
  ↓
Service 层（@Autowired 注入 + 数据转换）
  ↓
Controller（@ResponseBody + Jackson 自动 JSON）
  ↓
JSP + jQuery AJAX + ECharts（8 维度可视化）
```

> **核心变化**：老论文中数据清洗（MapReduce）是重点章节，新论文中数据已就绪，**弱化数据处理环节**，将主要内容放在 **SSM 框架整合与 Web 可视化实现** 上。

## 五、核心变更总结

### 5.1 架构层面

| 编号 | 变更项 | 说明 |
|------|--------|------|
| 1 | **零框架 → SSM 三层架构** | 引入 Spring + Spring MVC + MyBatis，从原生 Servlet 演进为标准企业级架构 |
| 2 | **无构建工具 → Maven** | 依赖版本集中管理，告别手动管理 jar 包的原始方式 |
| 3 | **手动 JDBC → Druid 连接池** | 数据库连接从 DriverManager 升级为阿里巴巴 Druid 连接池 |
| 4 | **手动 DAO → MyBatis Mapper** | 每表代码从 ~50 行手动 JDBC 减至 ~5 行接口 + 注解 |
| 5 | **硬编码配置 → 外部化配置** | 数据库连接信息从代码中抽离到 jdbc.properties |
| 6 | **手动 new → Spring IoC** | 对象创建与依赖关系交由 Spring 容器统一管理 |
| 7 | **手动拼 JSON → Jackson 自动序列化** | @ResponseBody 注解 + Jackson 自动完成对象到 JSON 的转换 |
| 8 | **两级 Spring 上下文** | Root Context 管理 Service/Mapper/数据源，Servlet Context 管理 Controller |

### 5.2 论文内容变化

| 编号 | 变更项 | 说明 |

| 10 | **SSM 框架成为核心** | 新论文第二章至第五章围绕 SSM 技术栈展开详细论述 |
| 11 | **代码展示现代化** | 老论文展示原生 JDBC/Servlet 代码，新论文展示注解式、声明式代码 |

## 六、结论

新项目（jobSSM）是在老项目（大一论文）基础上的 **框架化重构与论文升级**。老项目以"能跑就行"为目标，使用原生 Servlet + 手动 JDBC 实现，论文重点放在数据清洗流程（MapReduce/Hive/Sqoop）。新项目以 **SSM（Spring + Spring MVC + MyBatis）标准三层架构** 为核心，将数据清洗弱化为背景说明，论文重点转移到 **框架整合设计、分层架构实现、前后端数据交互与 ECharts 多维度可视化** 上。

这次迭代体现了从「大一入门级课程论文」到「毕业设计工程化论文」的成长路径：技术栈从业余走向专业，论文重心从数据处理转向软件架构。
