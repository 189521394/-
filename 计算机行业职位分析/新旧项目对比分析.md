## 二、技术栈对比

### 2.1 整体架构

| 层次 | 老项目 | 新项目 | 变更说明 |
|------|--------|--------|----------|
| **构建工具** | Gradle（`build.gradle`） | Maven（`pom.xml`） | 从 Gradle 迁移到 Maven，更符合 SSM 生态惯例 |
| **后端框架** | 原生 Jakarta EE（无框架） | **Spring 5.2.8 + Spring MVC 5.2.8 + MyBatis 3.5.2**（SSM） | 从零框架演进为标准 SSM 三层架构 |
| **数据库连接** | 手动 JDBC（`DriverManager`） | **Druid 1.1.20 连接池**（Spring 管理） | 引入连接池，提升性能与稳定性 |
| **ORM 层** | 手动编写 DAO 类 + 原生 SQL | **MyBatis 注解式 Mapper**（`@Select`） | 从手动 JDBC 到声明式 ORM |
| **依赖注入** | 手动 `new` 对象 | **Spring IoC**（`@Autowired`） | 引入控制反转，降低耦合 |
| **JSON 处理** | `org.codehaus.jettison` | **Jackson 2.13.4** | 使用主流 JSON 库 |
| **事务管理** | 无 | **spring-tx 5.2.8** | 引入声明式事务支持 |

### 3.2 数据访问层（DAO/Mapper）对比

**老项目** — 手动 DAO 模式（文档示例）：
```java
// cn/MyCn.java — 手动管理 JDBC 连接
public class MyCn {
    private Connection cn;
    public Connection getConnection() {
        Class.forName("com.mysql.cj.jdbc.Driver");
        String url = "jdbc:mysql://localhost:3306/job?...";
        cn = DriverManager.getConnection(url, "root", "114514");
        return cn;
    }
    public void close() { cn.close(); }
}

// dao/XxxDao.java — 手动编写 SQL + 结果集映射
```

**新项目** — MyBatis 注解式 Mapper：
```java
@Mapper
public interface CityMapper {
    @Select("select * from city_count order by city_count desc")
    public ArrayList<CityCountBean> getCityCountAll();
}
```

> **关键改进**：
> - 连接管理从手动 `DriverManager` → Druid 连接池（Spring 托管）
> - SQL 执行从手动 `Statement/ResultSet` → MyBatis 注解自动映射
> - 无需手动关闭连接（连接池自动回收）
> - 代码量大幅减少（每个 Mapper 仅需接口 + 注解）

### 3.3 业务层（Service）对比

**老项目**：手动实例化 DAO，手动管理事务

**新项目**：
```java
@Service
public class CityService {
    @Autowired
    private CityMapper cityMapper;   // Spring IoC 自动注入
    
    public DataBean[] getCityCountAll() {
        ArrayList<CityCountBean> cityCountAll = cityMapper.getCityCountAll();
        // ... 转换为统一的 DataBean 返回
    }
}
```

> **关键改进**：
> - `@Service` + `@Autowired` 替代手动 new 对象
> - 业务逻辑与老项目保持一致（ArrayList → DataBean[] 转换）

## 五、数据流对比

### 老项目数据流

```
MySQL ← Sqoop ← Hive ← HDFS ← MapReduce ← 原始数据
  ↓
手动 JDBC（MyCn.java）
  ↓
DAO 层（手动 SQL）
  ↓
Service 层（手动组装）
  ↓
Controller（手动 JSON 序列化）
  ↓
JSP + jQuery Ajax + ECharts
```

### 新项目数据流

```
MySQL ← Sqoop ← Hive ← HDFS ← MapReduce ← 原始数据
  ↓
Druid 连接池（Spring 托管）
  ↓
MyBatis Mapper（注解 SQL）
  ↓
Service 层（Spring 注入）
  ↓
Controller（Jackson 自动 JSON）
  ↓
JSP + jQuery Ajax + ECharts
```

> **说明**：新旧项目在**数据处理阶段**（Hadoop MapReduce 清洗 → Hive 分析 → Sqoop 导出到 MySQL）完全一致，重构仅涉及**数据可视化阶段**的后端架构。

## 六、核心变更总结

### 6.1 架构层面

| 编号 | 变更项 | 说明 |
|------|--------|------|
| 1 | **零框架 → SSM 框架** | 引入 Spring + Spring MVC + MyBatis 标准三层架构 |
| 2 | **Gradle → Maven** | 构建工具统一到 Maven，依赖管理更规范 |
| 3 | **手动 JDBC → Druid 连接池** | 数据库连接从手动管理升级为连接池，性能提升 |
| 4 | **手动 DAO → MyBatis Mapper** | 数据访问从手写 JDBC 代码升级为接口 + 注解声明式 |
| 5 | **硬编码配置 → 外部化配置** | 数据库连接信息从代码中抽离到 `jdbc.properties` |
| 6 | **手动 new → IoC 容器** | 对象创建与生命周期管理交由 Spring 容器 |

### 6.2 代码质量

| 编号 | 变更项 | 说明 |
|------|--------|------|
| 7 | **代码量减少** | DAO 层每个表从 ~50 行手动 JDBC → ~5 行 Mapper 接口 |
| 8 | **耦合度降低** | 通过接口 + 依赖注入，各层之间松耦合 |
| 9 | **可测试性提升** | Spring 托管 Bean 便于单元测试与 Mock |
| 10 | **包路径规范化** | `bean/` → `org.bean/`，符合 Maven 项目标准 |

### 6.3 运维与扩展性

| 编号 | 变更项 | 说明 |
|------|--------|------|
| 11 | **依赖版本管理** | `pom.xml` 集中管理所有依赖版本，可追溯、可复现 |
| 12 | **事务支持** | `spring-tx` 提供声明式事务，老项目无事务保障 |
| 13 | **连接池监控** | Druid 提供 SQL 监控、性能统计等运维能力 |
| 14 | **可扩展性** | SSM 框架易于扩展（如添加拦截器、AOP、全局异常处理等） |

### 6.4 保持不变

| 编号 | 保留项 | 说明 |
|------|--------|------|
| 1 | **前端完全一致** | JSP + jQuery + ECharts + view.js + CSS 均未改动 |
| 2 | **API 接口一致** | 8 个数据接口路径（`/city`, `/degree`, `/experience` 等）保持不变 |
| 3 | **数据库结构一致** | MySQL 中 8 张分析结果表结构不变 |
| 4 | **业务流程一致** | 数据预处理（MR）→ 数据分析（Hive）→ 数据导出（Sqoop）→ 可视化（Web） |
| 5 | **Bean 类结构一致** | 9 个 Bean 类的字段与方法完全保留 |

## 七、结论

新项目（jobSSM）是对老项目（jobDataView）**数据可视化子系统的框架化重构**，核心目标是将原本基于**原生 Jakarta EE + 手动 JDBC** 的实现，升级为业界标准的 **SSM（Spring + Spring MVC + MyBatis）** 三层架构。

重构遵循了**前端不变、接口不变、数据不变**的原则（对外接口路径与返回格式完全兼容），在保持用户体验一致的前提下，大幅提升了代码的**可维护性、可扩展性和工程质量**。这是典型的从「能跑就行」到「工程化规范」的毕业设计迭代路径。
