# 电商实时数据分析系统 —— 论文大纲

## 第一章 绪论

1.1 选题背景与意义

随着电子商务的快速发展，电商平台每天产生海量的交易订单数据。传统的离线批处理方式（如Hadoop MapReduce）虽然能够处理大规模数据，但数据处理延迟较高，无法满足实时监控、实时决策的业务需求。基于此，本课题设计并实现一个基于Spark Streaming的电商实时大数据分析系统，通过Kafka消息队列采集模拟订单数据，利用Spark Streaming进行流式聚合计算，将结果实时存入Redis并通过WebSocket推送到前端ECharts可视化展示。研究意义在于：（1）探索流式大数据处理技术在电商场景中的应用；（2）为电商企业实时监控商品销售动态提供技术方案；（3）验证Spark Streaming + Kafka + Redis技术栈在实时分析场景下的可行性。

1.2 论文组织结构

本论文共分为七章。第一章绪论，介绍选题背景与研究内容；第二章相关技术介绍，阐述系统所使用的核心技术；第三章系统分析，进行需求分析与用例分析；第四章系统设计，包括架构设计、模块设计与数据模型设计；第五章系统实现，详述各模块的编码实现；第六章系统测试，对系统功能进行验证；第七章总结与展望，总结工作并指出改进方向。

## 第二章 相关技术介绍

2.1 Scala语言概述

Scala是一种运行于Java虚拟机（JVM）之上的多范式编程语言，融合了面向对象与函数式编程的特性。其语法简洁、表达能力强，支持高阶函数、模式匹配、不可变集合等函数式编程特性，同时与Java完全互操作，可直接调用Java类库。Scala是Apache Spark的原生开发语言，Spark核心API由Scala编写，使用Scala编写Spark应用程序代码量更少、执行效率更高，因此本系统的流处理模块采用Scala语言实现。

2.2 Apache Kafka消息队列

2.2.1 Kafka核心概念

Kafka是由Apache软件基金会开发的分布式消息队列系统，具有高吞吐、低延迟、可持久化的特点。Kafka的核心概念包括：Producer（消息生产者）负责向Kafka发送数据；Broker（消息代理）是Kafka的服务节点，负责存储和转发消息；Consumer（消息消费者）从Kafka拉取数据进行处理；Topic（主题）是消息的逻辑分类，Producer将消息发送到指定Topic，Consumer订阅Topic获取消息；Partition（分区）是Topic的物理分片，实现数据的并行读写与水平扩展。本系统中，模拟订单数据由Producer发送到名为"product_order"的Topic，Spark Streaming作为Consumer订阅该Topic进行实时消费。

2.2.2 Kafka在实时数据管道中的作用

在实时数据处理架构中，Kafka扮演着消息中间件的角色，连接数据生产者与数据消费者，起到解耦与缓冲的作用。数据生产者无需关心数据由谁消费、如何消费，只需将数据发送到Kafka即可；消费者按自身处理能力从Kafka拉取数据，避免数据涌入速度超过处理速度导致系统崩溃。此外，Kafka将消息持久化到磁盘并按时间窗口保留，消费者可以重复消费历史数据，使得数据重放与故障恢复成为可能。本系统利用Kafka构建了模拟订单生成（Producer端）到Spark Streaming实时计算（Consumer端）之间的数据传输通道。

2.3 Apache Spark Streaming流式计算引擎

2.3.1 Spark Streaming微批处理模型

Spark Streaming是Spark生态中的流式计算组件，其核心思想是将实时数据流按时间间隔切分为一系列小批次（Micro-batch），每个批次交由Spark引擎进行RDD处理，从而将流式计算转化为批处理。Spark Streaming提供DStream（离散化流）抽象，DStream本质上是一系列连续的RDD，每个RDD包含一个时间窗口内的数据。本系统设置批处理间隔为5秒，每5秒对Kafka中收到的订单数据进行一次聚合计算。

2.3.2 DStream与RDD操作

DStream支持map、filter、reduceByKey、groupByKey等转换操作（Transformation），以及foreachRDD、print等输出操作（Action）。转换操作用于对数据进行加工与聚合，输出操作用于将结果写入外部存储。本系统通过map操作解析JSON订单数据，通过groupByKey按商品品类分组聚合销售额，通过foreachRDD将聚合结果分区写入Redis。

2.3.3 Spark Streaming与Kafka的Direct集成模式

Spark Streaming与Kafka集成有两种模式：Receiver模式和Direct模式。Direct模式是官方推荐的集成方式，它直接从Kafka的分区读取数据，不依赖Receiver进程，具有更好的容错性和"恰好一次"的语义保证。本系统采用Direct模式（KafkaUtils.createDirectStream），Spark Streaming直接从Kafka Broker的Partition中拉取消息，消费偏移量由Spark自行管理。

2.4 Redis内存数据库

2.4.1 Redis概述与数据类型

Redis是一种基于内存的键值对数据库，数据直接存储在内存中，读写速度极快，广泛应用于缓存、实时统计、消息队列等场景。Redis支持多种数据结构，包括String（字符串）、Hash（哈希表）、List（列表）、Set（集合）、Sorted Set（有序集合）等。本系统使用Redis的Hash结构存储商品销售数据，Key为"orderTotal"，Field为商品品类名称，Value为该品类的累计销售额。

2.4.2 Jedis客户端与连接池

Jedis是Java语言操作Redis的官方客户端，提供了与Redis命令一一对应的API。为避免频繁创建和销毁TCP连接带来的性能开销，生产环境中通常使用JedisPool连接池来管理Redis连接。连接池预先创建一定数量的连接供应用复用，使用完毕后归还池中而非直接关闭。本系统在Scala端和Java端分别封装了JedisPool工具类，用于统一管理Redis连接的获取与释放。

2.5 WebSocket实时通信协议

WebSocket是一种基于TCP的全双工通信协议，客户端与服务器只需一次握手即可建立持久连接，此后双方可以随时互相发送数据。相比传统的HTTP轮询方式，WebSocket避免了频繁建立连接和传输HTTP头信息带来的开销，能够实现真正的服务端主动推送，延迟更低、通信效率更高。本系统使用WebSocket实现服务端到前端的实时数据推送：服务端每2秒从Redis读取最新销售额数据，通过WebSocket推送到浏览器，前端接收到数据后立即更新ECharts图表，实现商品销售额的实时可视化展示。

2.6 Spring MVC框架

Spring MVC是Spring框架中用于构建Web应用的表现层框架，基于MVC（Model-View-Controller）设计模式，提供前端控制器（DispatcherServlet）、处理器映射器、视图解析器等核心组件。开发者通过注解（如@Controller、@RequestMapping）即可完成请求路由与响应处理，无需编写繁琐的Servlet配置。本系统使用Spring MVC构建Web服务，通过@Controller定义页面控制器，将根路径"/"请求映射到index.jsp首页视图。

2.7 前端可视化技术（ECharts + jQuery）

ECharts是百度开源的数据可视化图表库，支持柱状图、折线图、饼图等多种图表类型，提供数据驱动、动态更新、交互式操作等特性。jQuery是一个轻量级JavaScript库，简化了DOM操作、Ajax请求等前端开发任务。本系统前端使用jQuery获取页面元素、创建WebSocket连接并解析服务端推送的JSON数据，使用ECharts绘制水平柱状图，通过调用setOption方法动态更新图表数据，实现商品销售额的实时可视化展示。

2.8 Maven项目管理工具

Maven是Apache软件基金会的项目构建与管理工具，通过pom.xml文件统一管理项目的依赖、编译、打包和部署流程。Maven采用约定优于配置的理念，提供标准的项目目录结构和生命周期（clean、compile、package、install等），并支持依赖传递解析，自动下载和管理项目所需的第三方类库。本系统使用Maven管理Spark、Kafka、Spring、Jedis等十余项依赖，通过pom.xml实现依赖版本统一控制和项目的自动化构建。

## 第三章 系统分析

3.1 可行性分析

3.1.1 技术可行性

本系统所采用的技术栈均为开源且业界广泛使用：Kafka具备成熟的高吞吐消息传输能力，Spark Streaming提供稳定的流式计算框架，Redis凭借内存读写特性适合实时数据存储，WebSocket协议在实时通信领域应用成熟，Spring MVC和ECharts均为经过大量项目验证的Web开发技术。开发团队具备Scala和Java编程基础，熟悉Linux服务器基本操作。上述技术社区活跃、文档完善，技术方案可行。

3.1.2 经济可行性

系统全部采用开源软件（Kafka、Spark、Redis、Spring、ECharts等），无商业授权费用。硬件方面，开发测试阶段可在虚拟机中搭建多节点集群，无需采购额外物理服务器，系统运行期间仅消耗服务器日常运维成本。整体开发与运行成本较低，经济上可行。

3.1.3 操作可行性

系统前端界面简洁直观，用户打开浏览器即可看到商品销售额实时柱状图，无需额外培训。后端服务启动后，数据模拟生成、流式聚合、实时推送均由系统自动完成，运维人员只需掌握基本的集群启停操作。系统操作简便，具有可行性。

3.2 功能需求分析

3.2.1 模拟订单数据生成

系统需持续生成模拟电商订单数据，以验证实时数据处理链路。每条订单包含唯一标识（UUID）、商品品类名称、随机价格三个字段。商品品类覆盖新鲜果蔬、肉禽水产、粮油干货、休闲零食、乳制品、日化洗护、速冻食品、酒水饮料、厨房调味、针织家居共十个品类。订单以每秒一条的频率生成，以JSON格式发送至Kafka的product_order主题，模拟真实电商环境下的订单流。

3.2.2 订单数据实时采集与传输

系统需通过Kafka消息队列实现订单数据的实时采集与可靠传输。Kafka Topic负责暂存模拟生成的订单消息，Spark Streaming作为消费者以Direct模式从Kafka订阅数据，按5秒微批次间隔拉取消息进行处理。数据流转需保证低延迟，从订单生成到进入流计算环节的时间控制在秒级。

3.2.3 商品销售额实时聚合统计

系统需对采集到的订单数据按商品品类进行实时聚合计算：解析每条订单的JSON数据，提取品类名称和价格字段，按品类分组并对销售额求和，完成各品类累计销售额的实时统计。计算结果写入Redis的Hash结构（Key为orderTotal），使用原子累加命令实现销售额的持续更新，确保并发场景下数据准确。

3.2.4 统计结果实时可视化展示

系统需将Redis中存储的销售额数据以图表形式在前端页面实时展示。后端服务定时从Redis读取各品类最新销售额数据并序列化为JSON，通过WebSocket主动推送到浏览器前端（间隔2秒）。前端页面使用ECharts绘制水平柱状图，横轴为销售额，纵轴为品类名称，接收到推送数据后动态更新图表，实现商品销售额的实时可视化监控。

3.3 非功能需求分析

系统需满足以下非功能性要求：（1）实时性：从订单数据生成到前端图表更新的端到端延迟控制在秒级；（2）可靠性：Kafka消息持久化不丢失，Spark Streaming支持故障恢复，确保系统长时间稳定运行；（3）可扩展性：Kafka集群可按需增加Broker节点，Spark可通过增加分区数提升并行处理能力；（4）易用性：前端页面布局清晰，数据图表自动刷新，用户无需任何操作即可观察实时销售动态。

3.4 系统用例分析

本系统的参与者包括两类角色：系统管理员负责启动Kafka集群、启动Spark Streaming流计算任务、启动Web应用服务，并监控系统各环节运行状态；普通用户通过浏览器访问系统首页，查看商品销售额实时柱状图，图表数据自动更新，无需手动刷新页面。

4.2.1 模拟数据生成模块

模拟数据生成模块负责持续产生模拟电商订单数据，为整个实时处理链路提供数据源。该模块包含三个子功能：随机订单生成、JSON序列化与Kafka消息发送。

在随机订单生成子功能中，系统预先定义十个商品品类（新鲜果蔬类、肉禽水产类、粮油干货类、休闲零食类、乳制品类、日化洗护类、速冻食品类、酒水饮料类、厨房调味类、针织家居类），每次生成订单时随机选取一个品类，并生成0至499之间的随机价格，订单唯一标识采用UUID去除短横线的方式生成。订单数据以每秒一条的频率持续产生，模拟真实电商环境下用户不断下单的行为特征。

JSON序列化子功能负责将订单对象转换为JSON格式字符串。Kafka消息队列以字符串形式传输数据，因此需要将Product案例类对象序列化为标准JSON格式，便于下游消费者解析处理。

Kafka消息发送子功能负责将序列化后的JSON订单数据发送至Kafka集群。Producer的acks参数设置为"all"，确保消息在所有Broker确认接收后才认为发送成功，保证数据的可靠性。目标Topic为"product_order"，该Topic作为数据生成层与流式计算层之间的数据通道，后续由Spark Streaming订阅消费。

4.2.2 流式计算处理模块

流式计算处理模块是本系统的核心模块，负责从Kafka实时消费订单数据并进行聚合计算。该模块包含四个子功能：Kafka数据消费、JSON解析与字段提取、按品类分组聚合、分区并行写入Redis。

Kafka数据消费子功能采用Spark Streaming的Direct模式（KafkaUtils.createDirectStream）从Kafka订阅"product_order"主题的数据。与Receiver模式相比，Direct模式直接从Kafka分区的物理存储中拉取数据，不依赖额外的Receiver进程，具有更好的容错性和"恰好一次"的消费语义。批处理间隔设置为5秒，即每5秒将本时间段内接收到的订单消息汇集为一个微批次进行处理，在实时性与计算吞吐量之间取得平衡。

JSON解析与字段提取子功能对每条订单的JSON字符串进行解析，提取品类名称（name）和订单价格（price）两个关键字段，并将其转换为（品类，价格）的键值对形式，为后续分组聚合做准备。

按品类分组聚合子功能是流式计算的核心逻辑。首先通过groupByKey操作将同一品类的所有订单分组，然后使用reduceLeft函数对组内价格列表进行求和，得到每个品类在当前微批次内的销售额合计。聚合结果以（品类，批次销售额）的键值对形式输出。

分区并行写入Redis子功能通过foreachRDD操作将聚合结果写入Redis。为保证写入性能，采用foreachPartition方式以分区为单位获取Jedis连接——每个分区复用同一个连接完成该分区内所有数据的写入，避免了逐条获取和释放连接的开销。写入时调用Redis的hincrByFloat命令，以原子方式将批次销售额累加到对应品类的累计值上，确保并发场景下数据的一致性。

4.2.3 数据存储模块

数据存储模块负责管理系统与Redis之间的数据交互，为实时统计结果提供存储与读取能力。该模块包含两个子功能：Jedis连接池管理与Redis Hash累加存储。

Jedis连接池管理子功能负责封装Redis连接的获取与释放逻辑。由于Redis连接基于TCP协议，频繁创建和销毁连接会带来显著的性能开销，因此系统分别在Scala端和Java端封装了JedisPool工具类。Scala端的redisUtil单例对象供Spark Streaming流处理任务使用，在进程启动时加载redis.properties配置文件并初始化连接池；Java端的redisClient类供Web应用使用，通过静态初始化块完成连接池的创建与参数配置。连接池参数包括最大连接数（maxTotal）、最大空闲连接数（maxIdle）、最小空闲连接数（minIdle）以及连接等待超时时间（maxWaitMillis），通过合理设置这些参数，可在连接复用率与资源占用之间取得平衡。

Redis Hash累加存储子功能定义了统计结果的存储结构。系统选用Redis的Hash数据类型，Key命名为"orderTotal"，其中每个Field存储一个品类名称，对应的Value存储该品类的累计销售额。Hash结构的优势在于：同一Key下的多个Field共享相同的过期策略与命名空间，便于数据的统一管理；hincrByFloat命令以原子方式完成"读取现有值→加上新增值→写回"的完整操作，避免了读写分离带来的并发竞态问题，确保多个微批次并行写入时数据的正确性。

4.2.4 Web可视化与实时推送模块

Web可视化与实时推送模块负责将Redis中的统计结果以可视化的形式呈现给用户，并实现数据的实时更新。该模块包含三个子功能：Spring MVC页面路由、WebSocket实时推送与ECharts动态图表更新。

Spring MVC页面路由子功能负责接收浏览器请求并返回前端页面。通过@Controller注解定义页面控制器，将根路径"/"的HTTP请求映射到index.jsp首页视图。Spring MVC的视图解析器自动拼接前缀（/WEB-INF/views/）和后缀（.jsp），开发者只需返回逻辑视图名称"index"即可完成页面跳转。

WebSocket实时推送子功能是本模块的关键环节。服务端通过@ServerEndpoint注解将"/uiWebSocket"路径注册为WebSocket端点，浏览器通过该路径与服务端建立长连接。当首个客户端连接成功后，服务端启动定时任务，每隔2秒从Redis读取一次orderTotal中所有品类的销售额数据，序列化为JSON格式后通过WebSocket推送至所有在线客户端。采用2秒的推送间隔可在保证数据实时性的同时避免过于频繁的推送造成前端渲染压力。推送的数据格式为二维数组：[["品类1","品类2",...], ["销售额1","销售额2",...]]，分别存储品类名称数组与对应的销售额数组。

ECharts动态图表更新子功能运行在浏览器端。页面首次加载时初始化ECharts实例，配置水平柱状图的纵轴（品类名称）与横轴（销售额）。当通过WebSocket接收到新的销售数据后，调用setOption方法更新图表的data属性，ECharts自动执行动画过渡效果，实现图表的动态刷新，用户无需手动刷新页面即可持续观察各品类销售额的实时变化。

5.2 模拟订单数据生成模块实现

模拟订单数据生成模块以 Scala 独立应用程序的形式运行，负责持续向 Kafka 发送模拟电商订单数据。该模块的实现涵盖三个核心环节：Kafka Producer 配置与初始化、随机订单生成算法、JSON 序列化与消息发送。完整代码位于 createData.scala 文件中。

5.2.1 Kafka Producer 配置与初始化

Kafka Producer 的配置通过 java.util.Properties 对象完成，各项参数以键值对形式设置。核心配置如下：

```scala
val prop = new Properties()
prop.put("bootstrap.servers", "hadoop1:9092,hadoop2:9092,hadoop3:9092")
prop.put("acks", "all")
prop.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer")
prop.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer")
val producer = new KafkaProducer[String, String](prop)
```

bootstrap.servers 参数指定 Kafka 集群中三个 Broker 节点的地址，Producer 通过该地址列表发现集群中的所有节点。acks 参数设置为 "all"，表示消息需要所有副本确认接收后才视为发送成功，这是最高级别的可靠性保证。Key 和 Value 的序列化器均设为 StringSerializer，因为订单数据在发送前已被序列化为 JSON 字符串。

5.2.2 随机订单生成算法

订单数据采用 Scala 的 case class 定义数据结构 Product(ID: String, name: String, price: Double)。随机订单的生成逻辑如下：

```scala
case class Product(ID: String, name: String, price: Double)

var products = Array("新鲜果蔬类","肉禽水产类","粮油干货类","休闲零食类",
    "乳制品类","日化洗护类","速冻食品类","酒水饮料类","厨房调味类","针织家居类")

val id: String = UUID.randomUUID().toString.replaceAll("-", "")
val index: Int = new Random().nextInt(products.length)
val name: String = products(index)
val price: Int = new Random().nextInt(500)
val product = Product(id, name, price)
```

订单 ID 由 Java 标准库的 UUID.randomUUID() 方法生成，并调用 replaceAll("-", "") 去除短横线，得到 32 位纯十六进制字符串作为唯一标识。品类名称从预设的十个品类数组中通过 Random.nextInt() 随机选取索引获得。价格通过 Random.nextInt(500) 生成 0 至 499 之间的随机整数，赋值给 Double 类型字段。订单生成后封装为 Product 案例类对象。

5.2.3 JSON 序列化与消息发送

Product 对象在发送前需转换为 JSON 字符串，系统使用 json4s-jackson 库完成序列化。Kafka ProducerRecord 指定目标 Topic 为 "product_order"，消息体为 JSON 字符串。订单以每秒一条的频率循环产生：

```scala
val json = new Json(DefaultFormats)
val proStr: String = json.write(product)
producer.send(new ProducerRecord[String, String]("product_order", proStr))
Thread.sleep(1000)
```

json.write(product) 方法将 Product 案例类对象序列化为 JSON 格式字符串，例如 {"ID":"abc123","name":"休闲零食类","price":168.0}。ProducerRecord 的构造函数接收 Topic 名称和消息值两个参数。producer.send() 方法异步发送消息，内部根据 Key 的哈希值将消息路由到对应 Partition。循环末尾的 Thread.sleep(1000) 控制发送频率为每秒一条。

以下是 createData.scala 的完整代码：

```scala
package r

import org.apache.kafka.clients.producer.{KafkaProducer, ProducerRecord}
import org.json4s.DefaultFormats
import org.json4s.jackson.Json

import java.util.{Properties, UUID}
import scala.util.Random

case class Product(ID: String, name: String, price: Double)

object createData {
    def main(args: Array[String]): Unit = {
        val prop = new Properties()
        prop.put("bootstrap.servers", "hadoop1:9092,hadoop2:9092,hadoop3:9092")
        prop.put("acks", "all")
        prop.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer")
        prop.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer")
        var products = Array("新鲜果蔬类","肉禽水产类","粮油干货类","休闲零食类",
            "乳制品类","日化洗护类","速冻食品类","酒水饮料类","厨房调味类","针织家居类")
        val producer = new KafkaProducer[String, String](prop)
        while (true) {
            val id: String = UUID.randomUUID().toString.replaceAll("-", "")
            val index: Int = new Random().nextInt(products.length)
            val name: String = products(index)
            val price: Int = new Random().nextInt(500)
            val product = Product(id, name, price)
            val json = new Json(DefaultFormats)
            val proStr: String = json.write(product)
            producer.send(new ProducerRecord[String, String]("product_order", proStr))
            Thread.sleep(1000)
        }
    }
}
```

5.3 Spark Streaming流式计算模块实现

Spark Streaming 流式计算模块是本系统的核心计算组件，负责从 Kafka 实时消费订单数据并完成按品类的销售额聚合。该模块以 Scala 应用程序形式运行，完整代码位于 processData.scala 文件中。实现涵盖四个步骤：StreamingContext 初始化与 Kafka Direct Stream 创建、数据解析与键值对转换、按品类分组聚合、分区并行写入 Redis。

5.3.1 StreamingContext 初始化与 Kafka Direct Stream 创建

Spark Streaming 应用程序的入口是创建 SparkConf 和 StreamingContext。系统在开发测试阶段以 local[*] 模式运行，利用本机所有 CPU 核心。批处理间隔设置为 5 秒：

```scala
val conf: SparkConf = new SparkConf()
    .setAppName("processData")
    .setMaster("local[*]")
val sc = new SparkContext(conf)
val ssc = new StreamingContext(sc, Seconds(5))
```

Kafka Direct Stream 的创建需要指定 Kafka 参数、Topic 列表以及位置策略和消费策略。Kafka 参数以 Map[String, Object] 形式传递：

```scala
val kafkaParams: Map[String, Object] = Map[String, Object](
    "bootstrap.servers" -> "hadoop1:9092,hadoop2:9092,hadoop3:9092",
    "key.deserializer" -> classOf[StringDeserializer],
    "value.deserializer" -> classOf[StringDeserializer],
    "group.id" -> "spark_product",
    "auto.offset.reset" -> "latest",
    "enable.auto.commit" -> (false: java.lang.Boolean)
)
val topics = Array("product_order")
val stream: InputDStream[ConsumerRecord[String, String]] = KafkaUtils.createDirectStream(
    ssc, LocationStrategies.PreferConsistent,
    ConsumerStrategies.Subscribe[String, String](topics, kafkaParams))
```

group.id 设置为 "spark_product"，标识该消费者所属的消费组。auto.offset.reset 设置为 "latest"，首次启动时从最新消息开始消费。enable.auto.commit 设置为 false，关闭自动提交偏移量，由 Spark Streaming 自行管理偏移量以保证精确一次（Exactly-Once）的消费语义。LocationStrategies.PreferConsistent 指示 Spark 尽量将分区均匀分配到各 Executor 上。

5.3.2 数据解析与键值对转换

从 Kafka 消费到的数据是 ConsumerRecord 对象，需要经过三次 map 转换提取出有效的（品类, 价格）键值对：

```scala
stream.map(cr => cr.value())
    .map(order => JSON.parseObject(order))
    .map(obj => (obj.getString("name"), obj.getDouble("price")))
```

第一个 map 操作从 ConsumerRecord 中提取 value() 字段，得到 JSON 字符串。第二个 map 操作调用 fastjson 的 JSON.parseObject() 方法将 JSON 字符串解析为 JSONObject 对象。第三个 map 操作从 JSONObject 中提取 "name"（品类名称）和 "price"（订单价格）字段，构造为 (String, Double) 二元组。

5.3.3 按品类分组聚合

聚合操作分为两步：首先用 groupByKey 将同一品类的所有订单分组，然后用 reduceLeft 对组内价格列表求和：

```scala
.groupByKey().map(t => (t._1, t._2.reduceLeft(_ + _)))
```

groupByKey 将键值对按 Key（品类名称）分组，同一品类的所有价格 Value 归并为一个 Iterable[Double] 集合。reduceLeft(_ + _) 对集合中的元素从左到右依次累加求和，得到该品类在当前 5 秒微批次内的销售额合计。聚合结果为 (品类名称, 批次销售额) 的键值对。

5.3.4 分区并行写入 Redis

聚合结果通过 foreachRDD 输出操作写入 Redis。为提高写入效率，内部采用 foreachPartition 以分区为单位批量写入：

```scala
.foreachRDD(rdd => {
    rdd.foreachPartition(orders => {
        orders.foreach(t2 => {
            val jedis: Jedis = redisUtil.pool.getResource
            jedis.hincrByFloat("orderTotal", t2._1, t2._2)
            jedis.close()
        })
    })
})
```

foreachPartition 为每个 RDD 分区执行一次闭包，闭包内部遍历该分区中的所有记录。每个分区复用一个 Jedis 连接（从 redisUtil.pool 连接池中获取），处理完毕后归还连接池，避免了逐条获取连接的开销。写入时调用 Redis 的 hincrByFloat 命令，参数依次为 Hash Key（"orderTotal"）、Field（品类名称）和增量值（批次销售额），以原子方式完成累加。

最后启动 StreamingContext 并等待终止：

```scala
ssc.start()
ssc.awaitTermination()
```

```scala
    def main(args: Array[String]): Unit = {
        val conf: SparkConf = new SparkConf()
            .setAppName("processData")
            .setMaster("local[*]")
        val sc = new SparkContext(conf)
        val ssc = new StreamingContext(sc, Seconds(5))
        val kafkaParams: Map[String, Object] = Map[String, Object](
            "bootstrap.servers" -> "hadoop1:9092,hadoop2:9092,hadoop3:9092",
            "key.deserializer" -> classOf[StringDeserializer],
            "value.deserializer" -> classOf[StringDeserializer],
            "group.id" -> "spark_product",
            "auto.offset.reset" -> "latest",
            "enable.auto.commit" -> (false: lang.Boolean)
        )
        val topics = Array("product_order")
        val stream: InputDStream[ConsumerRecord[String, String]] = KafkaUtils.createDirectStream(
            ssc, LocationStrategies.PreferConsistent,
            ConsumerStrategies.Subscribe[String, String](topics, kafkaParams))
        stream.map(cr => cr.value())
            .map(order => JSON.parseObject(order))
            .map(obj => (obj.getString("name"), obj.getDouble("price")))
            .groupByKey().map(t => (t._1, t._2.reduceLeft(_ + _)))
            .foreachRDD(rdd => {
                rdd.foreachPartition(orders => {
                    orders.foreach(t2 => {
                        val jedis: Jedis = redisUtil.pool.getResource
                        jedis.hincrByFloat("orderTotal", t2._1, t2._2)
                        jedis.close()
                    })
                })
            })

        ssc.start()
        ssc.awaitTermination()
    }
}
```

5.4 Redis数据存储模块实现

Redis 数据存储模块负责封装与 Redis 之间的连接管理和数据读写操作。由于系统的 Scala 端（Spark Streaming）和 Java 端（Web 服务）均需访问同一 Redis 实例，因此在两个语言端分别实现了 Jedis 连接池的封装。Scala 端封装供流处理任务使用，Java 端封装供 Web 服务使用。

5.4.1 Scala 端 JedisPool 封装（redisUtil.scala）

Scala 端采用单例对象（object）实现 JedisPool 的封装。单例对象在 JVM 中全局唯一，天然适合管理连接池这种全局共享资源：

```scala
package r

import org.apache.commons.pool2.impl.GenericObjectPoolConfig
import redis.clients.jedis.JedisPool

import java.util.Properties

object redisUtil {
    val prop = new Properties()
    prop.load(this.getClass.getClassLoader.getResourceAsStream("redis.properties"))
    val redisHost: String = prop.getProperty("jedis.host")
    val redisPort: Int = prop.getProperty("jedis.port").toInt
    val timeOut: Int = prop.getProperty("jedis.timeout", "30000").toInt
    val pool = new JedisPool(new GenericObjectPoolConfig(), redisHost, redisPort, timeOut)
}
```

redisUtil 对象在 JVM 首次加载时初始化，通过 ClassLoader 读取类路径下的 redis.properties 配置文件，提取主机地址、端口和超时时间参数，创建 JedisPool 实例。GenericObjectPoolConfig 使用默认的连接池参数配置。在 processData.scala 中，通过 redisUtil.pool.getResource 获取连接，使用完毕后调用 jedis.close() 归还。

5.4.2 Java 端 JedisPool 封装（redisClient.java）

Java 端采用静态初始化块实现 JedisPool 的封装，确保连接池在类加载时完成初始化且全局唯一：

```java
package org.example.dao;

import redis.clients.jedis.Jedis;
import redis.clients.jedis.JedisPool;
import redis.clients.jedis.JedisPoolConfig;

import java.io.IOException;
import java.util.Properties;

public class redisClient {
    private static Jedis jedis;
    private static JedisPool jedisPool;

    public static Jedis getJedis() {
        return jedisPool.getResource();
    }

    public static void closeJedis(Jedis jedis) {
        jedis.close();
    }

    private redisClient() { }

    static {
        Properties prop = new Properties();
        try {
            prop.load(redisClient.class.getClassLoader()
                .getResourceAsStream("redis.properties"));
            JedisPoolConfig poolConfig = new JedisPoolConfig();
            poolConfig.setMaxTotal(Integer.valueOf(prop.getProperty("jedis.max.total")));
            poolConfig.setMaxIdle(Integer.valueOf(prop.getProperty("jedis.max.idle")));
            poolConfig.setMinIdle(Integer.valueOf(prop.getProperty("jedis.min.idle")));
            poolConfig.setMaxWaitMillis(
                Long.valueOf(prop.getProperty("jedis.max.wait.millis")));
            String host = prop.getProperty("jedis.host");
            String port = prop.getProperty("jedis.port");
            jedisPool = new JedisPool(poolConfig, host, Integer.valueOf(port), 10000);
        } catch (IOException e) {
            throw new RuntimeException(e);
        }
    }
}
```

Java 端的 JedisPoolConfig 配置了更详细的连接池参数：最大连接数（maxTotal=60）、最大空闲连接数（maxIdle=30）、最小空闲连接数（minIdle=5）、最大等待时间（maxWaitMillis=30000）。连接池超时时间设置为 10000 毫秒。私有构造函数防止外部实例化，getJedis() 和 closeJedis() 作为静态方法提供统一的连接获取与归还接口。

5.4.3 Redis Hash 原子累加操作

聚合结果的写入使用 Redis 的 hincrByFloat 命令，该命令对 Hash 中指定 Field 的值进行浮点数累加。若 Field 不存在则先初始化为 0 再执行累加，整个过程是原子操作。调用方式如下：

```java
jedis.hincrByFloat("orderTotal", "休闲零食类", 168.0);
// 等效于 Redis 命令：HINCRBYFLOAT orderTotal 休闲零食类 168.0
```

在 Spark Streaming 的分区写入流程中，每个微批次的聚合结果通过 hincrByFloat 累加到对应品类的 Field 上，多个分区的并发写入由 Redis 单线程模型保证操作的串行化，不会出现数据竞争。

5.5.2 数据服务层（getData.java）

数据服务层负责从 Redis 读取 orderTotal 中的所有品类销售额数据，并序列化为前端可识别的 JSON 格式：

```java
package org.example.service;

import com.fasterxml.jackson.core.JsonProcessingException;
import com.fasterxml.jackson.databind.ObjectMapper;
import org.example.dao.redisClient;
import redis.clients.jedis.Jedis;

import java.util.Map;

public class getData {
    public String getInfo() {
        String result = "";
        Jedis jedis = null;
        try {
            jedis = redisClient.getJedis();
            Map<String, String> orders = jedis.hgetAll("orderTotal");
            Object[] objects = new Object[2];
            String[] products = new String[orders.size()];
            String[] prices = new String[orders.size()];
            Integer index = 0;
            for (Map.Entry<String, String> order : orders.entrySet()) {
                products[index] = order.getKey();
                prices[index] = order.getValue();
                index++;
            }
            objects[0] = products;
            objects[1] = prices;
            ObjectMapper mapper = new ObjectMapper();
            try {
                result = mapper.writeValueAsString(objects);
            } catch (JsonProcessingException e) {
                throw new RuntimeException(e);
            }
        } finally {
            if (jedis != null) {
                redisClient.closeJedis(jedis);
            }
        }
        return result;
    }
}
```

getInfo() 方法通过 redisClient.getJedis() 从连接池获取连接，调用 jedis.hgetAll("orderTotal") 获取 Hash 中所有 Field-Value 对。随后遍历 Map 将品类名称和销售额分别存入两个 String 数组，组合为 Object[2] 结构：objects[0] 为品类名称数组，objects[1] 为对应销售额数组。最后通过 Jackson 的 ObjectMapper.writeValueAsString() 将 Object 数组序列化为 JSON 字符串，格式为 [["品类1","品类2",...],["销售额1","销售额2",...]]。finally 块保证 Jedis 连接在任何情况下都能归还连接池。

5.5.3 WebSocket 服务端（UIwebSocket.java）

WebSocket 服务端使用 @ServerEndpoint 注解声明端点路径，通过定时任务向所有已连接客户端推送最新统计数据：

```java
package org.example.webSocket;

import org.example.service.getData;

import javax.websocket.OnClose;
import javax.websocket.OnMessage;
import javax.websocket.OnOpen;
import javax.websocket.Session;
import javax.websocket.server.ServerEndpoint;
import java.io.IOException;
import java.util.concurrent.CopyOnWriteArrayList;
import java.util.concurrent.Executors;
import java.util.concurrent.ScheduledExecutorService;
import java.util.concurrent.TimeUnit;

@ServerEndpoint("/uiWebSocket")
public class UIwebSocket {
    private static final CopyOnWriteArrayList<UIwebSocket> sockets =
        new CopyOnWriteArrayList<UIwebSocket>();
    private static final ScheduledExecutorService scheduler =
        Executors.newSingleThreadScheduledExecutor();
    private static volatile boolean pushStarted = false;
    private Session session;
    private getData getData = new getData();

    @OnOpen
    public void onOpen(Session session) {
        this.session = session;
        sockets.add(this);
        System.out.println("在线数量：" + sockets.size());
        startPush();
    }

    @OnClose
    public void onClose() {
        sockets.remove(this);
        System.out.println("在线数量：" + sockets.size());
    }

    @OnMessage
    public void onMessage(String message, Session session) { }

    private synchronized void startPush() {
        if (pushStarted) return;
        pushStarted = true;
        scheduler.scheduleWithFixedDelay(() -> {
            if (sockets.isEmpty()) return;
            String info = getData.getInfo();
            for (UIwebSocket socket : sockets) {
                try {
                    if (socket.session != null && socket.session.isOpen()) {
                        socket.session.getBasicRemote().sendText(info);
                    }
                } catch (IOException e) {
                    System.err.println("推送失败: " + e.getMessage());
                }
            }
        }, 0, 2, TimeUnit.SECONDS);
    }

    public void sendMessage(Session session, String message) {
        try {
            session.getBasicRemote().sendText(message);
        } catch (IOException e) {
            throw new RuntimeException(e);
        }
    }
}
```

@OnOpen 方法在客户端连接建立时触发，将当前 WebSocket 实例加入 sockets 列表，并调用 startPush() 启动定时推送任务。@OnClose 方法在连接关闭时将实例从列表中移除。startPush() 使用 synchronized 关键字保证推送任务只启动一次（pushStarted 标志位），通过 ScheduledExecutorService.scheduleWithFixedDelay() 以 2 秒为固定延迟持续执行推送任务。推送任务遍历 sockets 列表，调用 getData.getInfo() 获取最新 JSON 数据，通过 session.getBasicRemote().sendText() 向每个在线客户端发送。CopyOnWriteArrayList 保证遍历期间其他线程修改列表不会抛出并发修改异常。

```properties
logger.jersey.name = org.glassfish.jersey
logger.jersey.level = error
```

分别对 Spark（org.apache.spark）、Kafka（org.apache.kafka）、Zookeeper（org.apache.zookeeper）、Hadoop（org.apache.hadoop）和 Jersey（org.glassfish.jersey）的日志类别设置 ERROR 级别，只输出异常和错误信息，保持控制台输出清晰可读。
